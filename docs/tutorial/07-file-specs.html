Chapter 7 — File Specifications (F-specs)

↑ Table of contents

In this chapter

The F-spec (column 6 = F) declares every file the program uses. Each F-spec specifies the filename, whether it is input, output, or update; its designation (primary, secondary, full-procedural); its record format and length; the device it lives on (DISK, PRINTER, WORKSTN); and, for keyed files, where the key sits inside each record. The file is opened automatically when the program starts — there is no explicit open() call.

7.1 The "you already know" analogy

In a modern language you open a file with a mode and some metadata:

# Python
f = open("customers.dat", "rb")          # binary read
r = open("report.txt", "w")              # text write
m = open("master.dat", "r+b")            # read-write

You tell the runtime the name, the mode (read, write, read-write), and implicitly the device (disk). RPG II folds all of that into a single declarative line — the F-spec — but with two extra twists: the designation (how the program cycle interacts with the file) and the device (which peripheral the file lives on).

This is familiar ground. An F-spec is a declarative open() call. You write it once at the top of the program; the runtime opens the file before the first line of logic runs and closes it when the program ends. What is different is that you also declare how the cycle drives the file — whether it reads records for you automatically (primary) or leaves you in charge (full-procedural).

7.2 F-spec column map

Every F-spec line has an F in column 6. The remaining columns fall into logical groups:

FieldColsDescription
Filename7–14The file's name (also used as the on-disk filename)
File Type15I (input), O (output), U (update), C (combined)
File Designation16P (primary), S (secondary), F (full-procedural), etc.
File Format19F (fixed-length, the common case)
Record Length24–27Number of bytes per record
Mode of Processing28Blank (sequential), L (limits), R (random)
Key Length29–30Length of the key field (for indexed files)
Record-Addr Type31A (zoned key), I (RRN), P (packed key), blank
Organization32I (indexed), T (address-output), blank (sequential)

Columns 17–18, 20–23, and the gaps are reserved or rarely used. For most programs you only need the filename, type, designation, format, record length, and device — everything else can stay blank.

7.3 File type (col 15) — the "mode"

Column 15 is the closest thing to a mode argument. It tells the runtime which direction data flows:

CodeModeModern analogyMeaning
IInput"r"Read-only. The cycle (or CHAIN/READ) reads records from this file.
OOutput"w"Write-only. The program produces records via O-specs. The file is created or truncated.
UUpdate"r+"Read and rewrite in place. You can read a record, modify it, and write it back.
CCombined—Both input and output on the same file (rare).

7.4 File designation (col 16) — how the cycle treats the file

This is the column with no modern equivalent. It tells the program cycle who is responsible for reading this file:

CodeDesignationCycle behaviour
PPrimaryThe cycle reads one record per iteration, automatically. Most input files are primary. There can be only one primary file (or one matched set).
SSecondaryThe cycle also reads this file, merging it with the primary by match field (M1). See Chapter 19.
FFull-proceduralThe cycle does not read this file. You control access entirely through CHAIN, READ, SETLL, etc. See Chapter 18.

This is familiar ground. Think of a primary file as a for line in file: loop that the language runs for you. A full-procedural file is one you manage yourself — you call read() when you want the next record. The designation column is what switches between those two models.

7.5 Device (cols 40–46)

The device column tells the runtime what kind of peripheral the file lives on. In practice there are four you will encounter:

DeviceWhat it meansTypical file type
DISKA file on disk. The most common device for input, output, and update.I, O, U
PRINTEROutput to a printer (mapped to a text file in modern systems). Used for reports.O
WORKSTNInteractive workstation display for screen-based I/O.C, U
SPECIALA user-supplied I/O routine — a hook for custom input/output logic.any

The device name is left-justified starting at column 40. DISK and PRINTER together cover the vast majority of batch programs.

7.6 Record length and format

The record length (cols 24–27) declares the fixed number of bytes per record. The runtime reads exactly this many bytes per record (with newline-tolerance: if a newline arrives before the full length, the remainder is padded with spaces).

Format (col 19) is almost always F (fixed-length). A fixed-length file with 80-byte records is the bread-and-butter of RPG II batch processing — it mirrors the 80-column punched card. Printer files typically use 132 to match a standard line-printer width:

7.7 Keyed files

An indexed (keyed) DISK file declares its key in four F-spec columns:

FieldColsPurpose
Key Length29–30Number of bytes in the key field
Record-Addr Type31A (zoned key), I (relative record number), P (packed key)
Organization32I for indexed (keyed)
Key Start35–381-based byte position where the key begins in each record

Together these four fields let the runtime build (or open) an index so you can do random access via CHAIN and positioned sequential access via SETLL/READ/READE — all covered in Chapter 18.

7.8 Common patterns

Most programs use one of a handful of F-spec combinations. Here are the four you will see over and over:

Line by line:

  • Primary sequential input — IP means input + primary. The cycle reads one record per iteration. This is the default starting point for most programs.
  • Update file — UF means update + full-procedural. You read records with READ or CHAIN, modify them in C-specs, and write them back via an update O-spec (UPDATE in cols 16–18).
  • Printer output — O means output, device is PRINTER. The record length is the print line width (typically 132). O-specs describe what prints.
  • Keyed full-procedural — IF means input + full-procedural. You control access with CHAIN, SETLL, READ, READE. The key columns (2AI and 1) define the index.

What's next

F-specs declare your files, but they say nothing about what is inside each record. That is the job of the I-spec — the input specification that maps byte positions to named fields. If the F-spec is the open() call, the I-spec is the struct that describes the record layout.

That is Chapter 8 — Input Specifications.
