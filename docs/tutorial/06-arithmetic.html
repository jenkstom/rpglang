Chapter 6 — Arithmetic Operations

↑ Table of contents

In this chapter

RPG II provides eight arithmetic operations: ADD, SUB, MULT, DIV, MVR (move remainder), Z-ADD (zero-add / assign), Z-SUB (negate), and SQRT (square root). All operate on the scaled-integer numeric model from Chapter 5: the compiler automatically aligns decimal points between operands, adjusts result scales, and optionally rounds via half-adjust. There is no operator-overloading or expression syntax — every operation is a single named opcode on one calculation line.

6.1 The "you already know" analogy

In a modern language you write total = a + b — one expression with an operator and an assignment. RPG II has no expression syntax. There is no + operator, no =, and no operator precedence. Instead, each arithmetic act is a single named operation on one calculation line, with its operands slotted into fixed columns.

You already know this. Think of each arithmetic C-spec line as a single assembly instruction: ADD source1 source2 → destination. You compose multi-step calculations by chaining lines, one operation at a time, top to bottom — exactly the way you would sequence statements, just without infix operators.

The operands occupy the same columns on every calculation line: Factor 1 (cols 18–27), the operation (28–32), Factor 2 (33–42), and the result field (43–48). So ADD AMT TOTAL — with Factor 1 left blank — means "add AMT into TOTAL":

....+....1....+....2....+....3....+....4....+....5....+....6....+....7....+....8
     C                     ADD  AMT       TOTAL

There is no ambiguity about which operand is the source and which is the destination: position decides. Factor 2 is always the right-hand operand; the result field is always where the answer lands.

The eight operations and what each computes:

Two operations deserve a quick orienting note before we go deeper. Z-ADD is the assignment operation: it clears the result field and stores Factor 2 into it (the "Z" is for "zero" — it zeroes the field first, which is why it is the canonical way to initialise a variable). Z-SUB does the same but negates. Both ignore Factor 1 entirely:

....+....1....+....2....+....3....+....4....+....5....+....6....+....7....+....8
     C                     Z-ADD42        X            ← X = 42
     C                     Z-ADD0         TOTAL        ← TOTAL = 0 (initialise)

6.3 Factor 1 omission — the accumulator pattern

For ADD, SUB, MULT, and DIV, Factor 1 is optional. When you leave it blank, the result field is used as both the source and the destination. This is the heart of the accumulator pattern:

....+....1....+....2....+....3....+....4....+....5....+....6....+....7....+....8
     C                     ADD  AMT       TOTAL        ← TOTAL = TOTAL + AMT
     C                     ADD  5         COUNT        ← COUNT = COUNT + 5

Because Factor 1 is blank, ADD AMT TOTAL reads as "TOTAL = TOTAL + AMT" — the result field feeds back into itself. Across the program cycle, where every line runs once per input record, this builds a running total automatically:

If you do supply Factor 1, the result field's old value is ignored — the result is computed purely from the two factors. So ADD A B SUM means SUM = A + B, overwriting whatever SUM held:

....+....1....+....2....+....3....+....4....+....5....+....6....+....7....+....8
     C           A         ADD  B         SUM          ← SUM = A + B (F1 present)

6.4 Decimal-point alignment

Because every numeric field is a scaled integer (Chapter 5), two fields can represent their values at different scales. A field with 2 decimals stores 1.50 as the integer 150; a field with 0 decimals stores the value 2 as the integer 2. Adding those raw integers (150 + 2 = 152) would be nonsense — you would be adding 1.50 to 0.02.

RPG II avoids this by automatically aligning operands at the implied decimal point before it computes. When the scales differ, the compiler rescales the lower-precision operand up to match. Adding a 0-decimal field (2) to a 2-decimal field (1.50):

  • 2 stored as 2 (0 decimals) → scaled up to 2 decimals: 200
  • 1.50 stored as 150 (2 decimals) → already at 2 decimals
  • 200 + 150 = 350, which at 2 decimals is 3.50

MULT and DIV have their own scale rules. MULT's result decimal count is the sum of the operands' decimal counts (multiplying a 2-decimal field by a 1-decimal field yields a 3-decimal result). DIV scales the numerator up before dividing so that the quotient keeps its precision rather than being truncated to an integer. You control the final scale through the result field's declared decimal positions (column 52).

You already know this. This is exactly what a fixed-precision decimal type does: the runtime tracks each value's scale and lines up the decimal points for you. The difference is that in RPG II the scale is a fixed property of each field, declared once, and the compiler honours it on every operation.

6.5 Half-adjust (rounding)

Because arithmetic is exact but result fields have a fixed number of decimals, a result sometimes has more fractional digits than the result field can hold. By default those extra digits are truncated (silently dropped). To round instead, put an H in column 53 — the half-adjust flag.

Half-adjust rounds the result to the result field's decimal positions by adding 5 at the first dropped digit before truncating — standard "round-half-up" behaviour. It applies to ADD, SUB, MULT, and DIV; SQRT is always half-adjusted whether you ask for it or not.

The classic demonstration is integer division made precise. Dividing 100 by 7 into a 2-decimal result:

....+....1....+....2....+....3....+....4....+....5....+....6....+....7....+....8
     C           100       DIV  7         Q        2     ← truncates → 14.28
     C           100       DIV  7         Q        2H    ← half-adjust → 14.29

Both lines are identical except for the H in column 53. Without it the third decimal (the .857… of 14.2857…) is dropped, giving 14.28; with it, that dropped digit rounds the result up to 14.29.

You already know this. Half-adjust is the language's per-operation round() — except you switch it on with a single column flag rather than wrapping the expression in a rounding call.

6.6 DIV and MVR (quotient and remainder)

DIV stores the quotient in its result field. The remainder is a separate quantity, and to capture it you must follow the DIV immediately with an MVR (move remainder) operation, which stores the remainder into its own result field:

....+....1....+....2....+....3....+....4....+....5....+....6....+....7....+....8
     C           100       DIV  7         Q            ← Q = 14 (quotient)
     C                     MVR            R            ← R = 2  (remainder)

The two operations are a pair: MVR has no factors of its own — Factor 1 and Factor 2 are blank — because it reads the remainder left behind by the immediately preceding DIV. If you want both the quotient and the remainder, you need both lines, always in that order. (When the DIV uses half-adjust for a fractional quotient, the MVR still reflects the integer remainder.)

6.7 Resulting indicators on arithmetic

Like every operation, arithmetic can set resulting indicators (cols 54–59) to record the sign of its result: HI (54–55) if the result is positive, LO (56–57) if negative, EQ (58–59) if zero. This lets you branch on the outcome of a computation without a separate COMP:

....+....1....+....2....+....3....+....4....+....5....+....6....+....7....+....8
     C                     ADD  5         T          50

Here 50 sits in the HI slot (cols 54–55), so indicator 50 turns ON when the result of the ADD is positive. A later line conditioned on 50 (or N50) then reacts to that sign — the same set-then-test pattern you met in Chapter 4, driven here by an arithmetic result rather than a comparison.

6.8 Overflow

Numeric fields cap at 15 digits. If a result exceeds its result field's declared length, the high-order digits are silently truncated — there is no exception, no overflow flag, and no warning. The value simply wraps around the field's capacity, losing its most significant digits.

You already know this. It is integer overflow — except that where many languages wrap modulo a power of two, RPG II truncates the leftmost digits and otherwise continues as if nothing happened. The practical lesson is the same as in any fixed-width numeric type: size your result fields generously, especially for MULT, whose result length is the sum of the operands' lengths.

6.9 Putting it together

The short program below exercises the pieces together: an accumulator built with Factor-1-omitted ADD, a DIV with half-adjust to compute an average, and a Z-ADD to copy the result. It averages three values (10, 25, 7) by summing them and dividing by 3:

....+....1....+....2....+....3....+....4....+....5....+....6....+....7....+....8
     C* Compute the average of three values: (10 + 25 + 7) / 3.
     C                     Z-ADD10        SUM
     C                     ADD  25        SUM
     C                     ADD  7         SUM
     C           SUM       DIV  3         AVG      2H
     C                     Z-ADDAVG       RPGRET   2
     C                     SETON                     LR

Trace the arithmetic: Z-ADD10 sets SUM to 10 (clearing it first). The next two lines add 25 and 7, accumulating SUM to 42 — all three ADDs omit Factor 1, so SUM feeds back into itself. DIV 3 AVG divides 42 by 3 into a 2-decimal result with half-adjust, giving AVG = 14.00. Z-ADDAVG copies that into RPGRET, and SETON LR signals the cycle to stop. Every step is one operation, one line, with the scaled-integer model quietly aligning decimals throughout.
