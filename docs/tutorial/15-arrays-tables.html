Chapter 16 — Arrays and Tables

↑ Table of contents

In this chapter

RPG II supports fixed-size arrays and a special "table" abstraction, both declared with the E-spec (Extension specification). Arrays are declared with a name, element count, element length, and decimal positions; they can be loaded with compile-time data (appended after the source) or prerun-time data (loaded from a file at startup). Element access uses ARRAY,INDEX syntax (1-based). Tables are arrays with a hidden "current element" pointer — LOKUP advances the pointer to the matched element, and a related table advances in lockstep. Operations include XFOOT (sum all elements) and LOKUP (search for a value). This chapter covers the E-spec, data loading, element access, tables vs arrays, and search/sum operations.

16.1 The "you already know" analogy

An RPG II array is a fixed-size array of numeric values — like int[] values = new int[5];. You cannot resize it or append to it; its size is fixed at compile time. An RPG II table is an array with an implicit iterator: LOKUP is like find() that not only tells you whether a match exists but also positions the "current element" pointer at it, like a database cursor.

This is familiar ground. An array is the ordinary indexed collection you know. A table is that same collection plus a hidden current-element cursor that LOKUP moves. And a pair of related tables that advance in lockstep is just a dictionary / map: look up a key in one, read the matching value from the other.

One detail surprises modern programmers: there is no separate "declare" syntax and "populate" syntax in the C-specs. The E-spec line declares the array, and its initial values come from data lines at the bottom of the source (compile-time) or from a file (prerun-time) — the same record-oriented thinking as the rest of RPG II.

16.2 The E-spec (Extension specification)

Arrays and tables are both declared on E-spec lines (column 6 = E). The only thing that distinguishes a table from an array is the name: a name beginning with TAB denotes a table (and so gets the hidden current-element pointer); anything else is an ordinary array. The E-spec columns are:

FieldColsDescription
Prerun-time filename11–18If present, load the array from this file at startup instead of compile-time data
Array/table name27–32The name. TAB-prefixed ⇒ table (has a current-element pointer)
Entries per record33–35How many elements appear on one compile-time/prerun-time data record
Number of entries36–39Total element count
Entry length40–42Digits (or characters) per element
Decimal positions440–9 for numeric; blank for alphanumeric
Sequence45A = ascending, D = descending, blank = unsorted
Alternating partner name46–51A second array/table whose elements interleave with this one
Partner entry length52–54Element length of the partner
Partner decimals56Decimal positions of the partner
Partner sequence57A/D for the partner

16.5 Element access: ARRAY,INDEX

In any factor or result field, write ARRAY,INDEX to access a specific element. The index is 1-based (the first element is index 1, not 0). The index may be a field name or a literal:

The first line searches array TABS (written TABS,ID at cols 33–39) for the value in KEY (factor 1, cols 18–20), using the field ID as the index, and sets indicator 10 (cols 58–59) on an equal match. On a match, ID is updated to the matched position. The second line runs only when indicator 10 is ON (conditioning at cols 9–10).

For tables, a bare name (no ,INDEX) resolves to the current element — the one LOKUP last positioned. TABNUM,3 still works as an explicit array-style reference; the bare TABNUM reads whatever element the cursor currently points at.

16.6 XFOOT — sum an array

XFOOT ARRAY RESULT sums all elements of the array into the result field. The analogy: result = sum(array) or array.Sum(). We have already used it twice above. It also sets resulting indicators (HI/LO/EQ) based on the sign of the sum, so you can test whether the total is positive, negative, or zero in the same operation.

16.7 LOKUP — search an array or table

LOKUP searches the array/table named in factor 2 for a value matching factor 1, and reports the outcome through three resulting indicators (cols 54–59):

Indicator slotColsTurned ON when…
HI54–55An element greater than the key exists
LO56–57An element less than the key exists
EQ58–59An exact match is found

The simplest case is an exact-match search on a table. Here TABNUM holds [10, 20, 30, 40, 50]; we look up 30 and, on a match, read the current element:

Walk through it: Z-ADD 30 puts 30 into KEY. LOKUP searches TABNUM for 30, finds it at element 3, sets indicator 10 (the EQ slot, cols 58–59), and — because TABNUM is a table — advances its current-element pointer to element 3. The next line, conditioned on indicator 10, does Z-ADD TABNUM: the bare table name reads its current element (30) into RPGRET.

The A at col 45 of the E-spec marks TABNUM as ascending, which lets a HI/LO search find the nearest qualifying element rather than merely "any" one. For an unindexed array, write ARRAY,IDX in factor 2; on a match the index field is updated to the position found.

16.8 Related tables — a built-in dictionary

Two tables declared as alternating partners advance in lockstep: a LOKUP that positions one also positions the other. This is a dictionary lookup — find a key in the first table, read the corresponding value from the second. Declare the partner in cols 46–57 of the E-spec, and name it in the result field of the LOKUP:

The E-spec declares TABKEY (3 entries, 2 digits) with alternating partner TABVAL at cols 46–51 (3 entries implied, 3 digits, 0 decimals). The compile-time data interleaves the two tables element by element: TABKEY=10, TABVAL=100, TABKEY=20, TABVAL=200, TABKEY=30, TABVAL=300 — packed as 10 100 20 200 30 300.

The LOKUP searches TABKEY for 20, finds it at element 2, and — because the result field names the related table TABVAL — advances both current-element pointers to element 2. The bare name TABVAL then reads its current element, 200, into RPGRET. One search, two coordinated cursors — exactly dict[key] → value.

16.9 MOVEA — bulk byte transfer

MOVEA copies bytes between an array and a field (or two fields) as a flat byte sequence, left-justified. It treats its operands as raw bytes rather than as typed values — useful for shuttling an array's contents into a single field or vice versa. See Chapter 14 for details.
