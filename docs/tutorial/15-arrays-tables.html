Chapter 16 — Arrays and Tables

↑ Table of contents

Summary

RPG II supports fixed-size arrays and a special "table" abstraction via the E-spec (Extension specification). Arrays are declared with a name, element count, element length, and decimal positions; they can be loaded with compile-time data (appended after the source) or prerun-time data (loaded from a file at startup). Element access uses ARRAY,INDEX syntax (1-based). Tables are arrays with a hidden "current element" pointer — LOKUP advances the pointer to the matched element, and a related table advances in lockstep. Operations include XFOOT (sum all elements) and LOKUP (search for a value). This chapter covers the E-spec, data loading, element access, tables vs arrays, and search/sum operations.

Outline

16.1 The "you already know" analogy

  • An RPG II array is a fixed-size array of numeric values — like int[] values = new int[5];. You can't resize it or append to it.
  • An RPG II table is like an array with an implicit iterator — LOKUP is like find() that not only returns whether a match exists but also positions the "current element" pointer at the match, similar to a database cursor.
  • Related tables that advance in lockstep are like a pair of parallel arrays or a dictionary/map: look up a key in one, get the corresponding value from the other.

16.2 The E-spec (Extension specification)

  • The E-spec declares arrays and tables. Key columns:
    • Name (cols 27–32) — the array/table name. Names starting with TAB are treated as tables; others are arrays.
    • Number of entries (cols 36–39) — element count.
    • Entry length (cols 40–42) — digits per element.
    • Decimals (col 44) — decimal positions per element.
    • Prerun-time filename (cols 11–18) — if present, load from this file at startup instead of compile-time data.
    • Alternating/related table (cols 46–57) — partner table name and its length/decimals.

16.3 Compile-time data

  • Compile-time array data appears after the source code, introduced by a ** record:
** 
1020304050
  • The data is packed — elements are concatenated without delimiters. For 5 elements of length 5, the data line 1020304050 holds all five values.
  • This data is baked into the compiled program at build time.

16.4 Prerun-time data

  • If the E-spec specifies a filename in cols 11–18, the array is loaded from that file at program startup — before the cycle runs.
  • Useful when the data changes between runs but shouldn't require recompilation.

16.5 Element access: ARRAY,INDEX

  • In any factor or result field, write ARRAY,INDEX to access a specific element. The index is 1-based.
  • Example: NUMS,3 = the third element of NUMS. The index can be a field name or a literal.
  • For tables, a bare name (no ,INDEX) resolves to the current (shadow-selected) element. TABNUM,3 still works as an explicit array-style reference.

16.6 XFOOT — sum an array

  • XFOOT NUMS RESULT sums all elements of the array into the result field. The analogy: result = sum(nums) or nums.Sum().
  • Sets resulting indicators (HI/LO/EQ) based on the sign of the sum.

16.7 LOKUP — search an array or table

  • LOKUP KEY ARRAY INDEX searches the array for a value matching factor 1. Sets resulting indicators:
    • EQ — exact match found.
    • HI — a higher element found.
    • LO — a lower element found.
  • On a match, the index variable is updated to the matched position.
  • For tables, LOKUP on a bare table name advances the hidden current-element pointer to the match. A related table named in the result field advances in lockstep.
  • The analogy: indexOf() / find() / binary search, depending on the table's sequence flag.

16.8 MOVEA — bulk transfer

  • MOVEA copies bytes between an array and a field (or two fields) as a flat byte sequence. See Chapter 14.
