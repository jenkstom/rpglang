Chapter 3 — The Program Cycle

↑ Table of contents

In this chapter

The program cycle is the single most important concept in RPG II, and this chapter is devoted to it. When a program declares a primary input file, the language automatically wraps the program's calculation and output logic in an invisible loop: read a record → extract fields → run detail calculations → run total calculations on control breaks → repeat until end-of-file → run last-record (LR) totals → terminate. The programmer never writes while or readLine() — the cycle handles all of it. This chapter explains the cycle's phases, when each type of calculation and output fires, and how the LR (last-record) indicator signals the end. Without this mental model, RPG II programs appear to do magic; with it, everything clicks into place.

3.1 The "you already know" analogy

Think about the last time you wrote a program that produced a report from a file. No matter what language you used, the skeleton was almost certainly the same. You opened the file, then looped: read a record, pull the fields out of it, do whatever per-record work the job required, emit some output, and check whether more records remained. When the file ran out, you printed your grand totals, closed the file, and stopped. In pseudo-code it looks like this:

open file
while (record = read_next()):
    extract_fields(record)
    do_calculations()
    write_output()
write_totals()
close file

You wrote the while loop, the read call, the end-of-file test, and the total logic — every time, for every report. The loop is yours: you own it, you debug its off-by-one errors, and you are responsible for making sure the totals run exactly once at the end.

You already know this shape — it is the iterate-over-records loop you have written many times. Now consider what happens if the language itself supplies that entire outer loop for you, so that you never write it again. That is exactly what RPG II does.

In RPG II, the entire outer loop is implicit. You declare four things, and nothing else:

  • the file to read (an F-spec),
  • the record layout — which byte positions map to which field names (I-specs),
  • the per-record calculations — the logic that runs for each record (C-specs with a blank control level), and
  • the output format — what the report lines look like (O-specs).

The cycle supplies everything else: the open, the read, the field extraction, the loop-back, the end-of-file detection, and the final total-time pass. You write only the body and the totals. This is the core of the "report program generator" philosophy: the batch report pattern is so common that the language is the pattern.

The rest of this chapter explains exactly what the cycle does and when your code runs inside it. Timing is everything: in RPG II, a line of calculation can run once per record, once per group, or once at the very end — and which one happens depends on a single pair of columns you set when you write that line.

3.2 The cycle, step by step

Here is the full cycle as a flowchart. Read it top to bottom: the blue path is the per-record loop that runs for every input record; the red path is the end-of-file path that runs once at the very end. We will walk through each box in turn.

Step 1 — Open files

When the program starts, all files declared in F-specs are opened automatically. Input files are positioned at the first record; output files are created. You never write an open statement — the F-spec declarations are the open. This happens exactly once, before any record is read.

Step 2 — Read the next record

The cycle reads one record from the primary input file into an internal buffer. This is the equivalent of the read_next() call in the pseudo-code above — except you did not write it. If there are no more records (end-of-file), the cycle does not proceed to step 3; instead it branches right to the LR (last-record) path, which we cover in step 7 below.

Step 3 — Extract fields

The I-specs define which byte positions in the record map to which named fields. The cycle extracts them automatically — your code never touches the raw record buffer. By the time your calculations run, every field you declared is already populated with the current record's values and ready to use by name. This is the extract_fields(record) call from the pseudo-code, done for you.

Step 4 — Total time (control-level breaks)

Before detail processing, the cycle checks whether a control-level field — a field you declared with an L1–L9 marker on its I-spec — has changed value since the previous record. If it has, the corresponding control-level indicator turns on, and any C-spec conditioned by that level runs before detail processing for the new record. This is where subtotals get printed and accumulators get reset. If you are not using control levels (many simple programs do not), this step does nothing and you can ignore it. We cover control levels in depth in Chapter 15.

Step 5 — Detail time

C-specs with a blank control level — columns 7–8 left empty — run once per record. This is where most per-record logic lives: accumulating running totals, validating data, making decisions, setting indicators. Immediately after the detail calculations finish, the detail output (O-spec lines of type D) fires. This is the body of your loop — the do_calculations() and write_output() calls, driven for you once per record.

Step 6 — Loop back

After detail processing, the cycle returns to step 2 and reads the next record. This loop is automatic — there is no explicit goto, continue, or back-edge in your source. The dashed line in the diagram above is the loop-back, and it is drawn dashed precisely because you never write it.

Step 7 — Last record (LR)

When end-of-file is reached at step 2, the cycle does not process another record. Instead it turns on the LR (last-record) indicator and runs the total-time pass one final time. All control levels L1–L9 also turn on at LR, so the final group's subtotals print before any grand totals. Then any LR-conditioned calculations and output run — this is where you print grand totals. Finally all files close and the program terminates. This is your write_totals() and close file, supplied automatically.

3.3 Detail vs. total vs. LR timing

The cycle has three timing phases, and knowing when your code runs is the key to reading and writing RPG II. A single C-spec line can run once per record, once per group, or once at the very end — and the difference is just two columns. The diagram below shows exactly when each kind of calculation and output fires across a small set of records.

Follow the diagram left to right. Five records arrive, grouped by a key field into three groups: A (records 1–2), B (records 3–4), and C (record 5). Now watch when each kind of work fires:

  • Detail calculations run for every record — five green boxes, one per record. A detail C-spec is one whose control level (columns 7–8) is blank. This is where you accumulate per-record values, for example ADD AMT TOTAL, which adds each record's amount into a running sum.
  • Total calculations run only when a control field changes — a "break." In the diagram the key changes from A to B and from B to C, so the L1 total routine fires twice. A total C-spec is one whose control level is L1–L9. This is where you print subtotals and reset accumulators for the next group. (The deep dive on control levels is in Chapter 15.)
  • LR (last-record) calculations run once, at end-of-file. This is where you print grand totals. A C-spec conditioned by LR (in columns 7–8) runs here. Because all of L1–L9 also turn on at LR, the final group's subtotals print before the grand total — so nothing is lost.

You already know all three of these timings from the pseudo-code in section 3.1: detail time is the inside of the loop body; total time is the cleanup that runs after a group-by key changes; LR time is the after-the-loop code that runs once when the file is exhausted. RPG II just makes the distinction explicit in the source, via the control-level columns, instead of via where you physically place the code.

3.4 A complete program

Now let's see the cycle in action. Below is a complete, working RPG II program. It reads a file of sales records, each holding a three-digit AMT field, adds each amount into a running TOTAL, prints every amount on a detail line, and prints the grand total once at the end. This is the same job the pseudo-code in section 3.1 describes — but notice that the while loop, the read, and the end-of-file test are nowhere to be found.

Read it section by section, and map each piece onto a step of the cycle:

The F-specs (lines 1–2) declare the two files the cycle will open at step 1. SALES is an input file on disk — the IP in columns 15–16 means "input, primary," which is what tells the language that this file feeds the automatic read loop (step 2). REPORT is an output file to a printer. You do not open them; declaring them is enough. (F-specs are covered column by column in Chapter 7.)

The I-specs (lines 4–5) describe the record and its fields, which the cycle extracts at step 3. Line 4 identifies the record type; line 5 maps record positions 1 through 3 to the field name AMT. From now on, every time the cycle reads a record, those three bytes are automatically available as AMT. You never write the extraction code. (I-specs are covered in Chapter 8.)

The C-spec (line 7) is the entire detail calculation — step 5. Its control level (columns 7–8) is blank, so it runs once per record. It adds AMT into TOTAL. Because TOTAL is never reset, it accumulates across every record. This one line is the whole loop body. (C-specs are covered in Chapter 9.)

The O-specs (lines 9–14) describe the output, which the cycle emits at the right moment. The first record line (type D = detail) prints once per record, right after the detail calculation: the literal 'Amount =' followed by the value of AMT. The second record line (type T = total, conditioned on LR) prints once at the very end: the literal 'Grand total =' followed by TOTAL. You did not write "print" anywhere — you described what each line looks like and when it should appear, and the cycle emitted it. (O-specs are covered in Chapter 16.)

Tracing a run

Suppose the SALES file contains four records with amounts 010, 020, 005, and 007. Here is what the cycle does, step by step:

  1. Start: open SALES and REPORT.
  2. Record 1: read 010, extract AMT = 10, run ADD AMT TOTAL (TOTAL becomes 10), emit the detail line "Amount =      10". Loop back.
  3. Record 2: read 020, AMT = 20, TOTAL becomes 30, emit detail line. Loop back.
  4. Record 3: read 005, AMT = 5, TOTAL becomes 35, emit detail line. Loop back.
  5. Record 4: read 007, AMT = 7, TOTAL becomes 42, emit detail line. Loop back.
  6. End-of-file: no more records. Turn on LR. Run the total-time pass. The LR-conditioned total line fires once: "Grand total =      42". Close both files. Terminate.

Notice what you did not write: the open, the four reads, the loop-back, the end-of-file test, the LR turn-on, or the close. The cycle supplied all of it. Your source contributed the file declarations, the field map, one accumulation line, and two output descriptions — and that was enough to produce a complete report with a grand total.

3.5 When there is no primary file

Not every program reads files. A pure calculation program — a quick test of some arithmetic, a subroutine you want to exercise — might have only C-specs and no F-specs at all. What happens to the cycle then?

When there is no primary input file, the cycle degenerates: the C-specs run once, top to bottom, in a single pass. There is no read, no loop, and no automatic end-of-file. The program simply executes its calculations and then would run forever — unless you give it a way to stop.

The way to stop it is the LR indicator. You can turn LR on manually with the SETON operation, which is the RPG II equivalent of return or exit():

Here SETON sits in the operation columns (28–32), and LR names the indicator to turn on, written in the resulting-indicator columns (54–55). When this line executes, LR turns on, the cycle runs its final total-time pass, and the program terminates. In a program with a primary file you rarely need this — the cycle sets LR for you at end-of-file — but in a no-file program it is the only way to finish cleanly.

You already know this pattern. A no-file RPG II program is like a script with no input loop: it runs its body once and exits. SETON LR is the explicit return that a file-driven program gets for free at end-of-file.

3.6 The LR indicator

LR — "last record" — is the cycle's most important special indicator, and it deserves a section of its own because it appears everywhere. It turns ON automatically when the input file reaches end-of-file, and that single event drives the entire end-of-program sequence.

There are two ways LR turns on:

  • Automatically, when the cycle tries to read the next record and finds end-of-file instead. This is the normal case in any program with a primary input file. You do nothing — the cycle sets it for you.
  • Manually, when your code executes SETON LR. This is useful in no-file programs (as above) or when you want to stop early — for example, after finding the record you were looking for, or after hitting an error condition.

Once LR is on, two things happen, in order:

  1. The cycle runs the total-time pass one final time. Because every control level L1–L9 also turns on alongside LR, any pending group subtotals print first — so the last group is not left half-finished.
  2. Then LR-conditioned calculations and output run. This is where grand totals go. After that, all files close and the program ends.

The practical consequence is that LR is how you say "do this once, at the very end." You put LR in the control-level columns of a C-spec (7–8) or in the conditioning columns of an O-spec, and the cycle guarantees it runs exactly once, after every record has been processed. You saw this in the complete program above: the total output line carried LR in its conditioning columns, so it printed the grand total only at the very end.

LR is an indicator, and we treat the whole indicator system — conditioning, resulting, the general-purpose 01–99 flags, and how they wire calculations to output — in Chapter 4.

A mental shortcut

If you remember only one thing from this chapter, remember the timing rhyme:

detail = per record,   total = per group,   LR = once at end.

Every calculation and every output line in an RPG II program belongs to exactly one of those three moments. When you read a line, ask which moment it belongs to, and you will know when it runs.

What's next

You now understand the program cycle: the invisible read-process loop, the three timing phases, and the LR indicator that marks the end. This is the single biggest mental shift in learning RPG II, and everything that follows builds on it.

The cycle leans heavily on indicators — the LR flag is one, the control levels L1–L9 are others, and there are ninety-nine general-purpose flags besides. Indicators are how RPG II expresses conditions and records the outcomes of operations, and they are the subject of Chapter 4.
