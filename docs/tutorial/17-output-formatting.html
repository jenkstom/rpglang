Chapter 17 — Output Formatting

↑ Table of contents

In this chapter

RPG II formats numeric output through two mechanisms on the O-spec: edit codes (a single-letter code in column 38 that selects a predefined format for commas, decimal points, signs, and zero-suppression) and edit words (a quoted template string in columns 45–70 that gives you full control over the output picture). Edit codes are quick — like choosing a format specifier. Edit words are expressive — like writing a custom format template. This chapter covers all edit codes (1–4, A–D, J–Q), edit word syntax, and the zero-balance blank behavior that distinguishes adjacent code pairs.

17.1 The "you already know" analogy

Edit codes are like format specifiers — String.Format("{0:N2}", value) or Python's f"{value:,.2f}". One code selects a whole formatting pattern. Edit words are like custom format templates — "#,###.00". You draw the picture; the runtime fills in the digits.

This is familiar ground. An edit code is a named number format (with/without commas, with/without a sign). An edit word is a picture string where some characters are "fill in the digit here" and the rest are literals that print verbatim. Both live on the O-spec field line; the runtime applies them at print time.

17.2 Where formatting lives on the O-spec

Both mechanisms attach to a single O-spec field line (see Chapter 10 for the O-spec anatomy). The choice between them is positional: a non-blank single character in column 38 is an edit code; a quoted string beginning at column 45 is an edit word. You use one or the other on a given field line, not both:

MechanismColumnForm
Edit code38A single letter or digit (1–4, A–D, J–Q)
Edit word45–70A quoted template, e.g. ' , . '

In both cases the field name sits at cols 32–37 and the end position at cols 40–43, exactly as for an unformatted field. The formatting just changes how the value's digits are laid out within that end position.

17.4 The pairing rule — zero-balance blanking

The pairing rule. Adjacent codes differ only in zero-balance behavior. Code 1 and 2 are identical except that 2 blanks out a zero value entirely. The same relationship holds for 3/4, A/B, C/D, J/K, L/M, N/O, and P/Q. The odd code of each pair prints a zero as 0.00 (or 0); the even code prints nothing but blanks.

This matters for reports. A detail line where a quantity is genuinely zero looks cleaner with the zero suppressed (use the even code); a total line where you always want to show a figure, even zero, uses the odd code. Pick the pair for the formatting you want (commas? sign?), then pick odd-or-even for whether zero should be visible.

17.5 Edit words (cols 45–70) — custom templates

When col 38 is blank and cols 45–70 hold a quoted string, the string is an edit word — a template the runtime fills with the field's digits. The template's characters have special meanings:

CharacterMeaning
blankReplaceable position — filled by a source digit, right-aligned, zero-suppressed
, . / -Literal — printed verbatim (comma, period, slash, dash)
first 0Stops zero-suppression — digits from here on always print, even leading zeros
*Check protection — replaces leading zeros with asterisks instead of blanks
trailing - or CRSign indicator — printed only if the value is negative; blank otherwise
&Forces a literal blank in the output

The same mechanism drives dates ('  /  /  '), phone numbers, or any fixed layout where literal punctuation must sit between groups of digits. Because you draw the picture character by character, an edit word can express patterns no single edit code can.

17.6 Choosing edit codes vs edit words

Reach for an edit code whenever you want one of the standard number layouts — commas, a decimal point, an optional sign, and optional zero-suppression. It is one character and covers the overwhelming majority of report formatting. Reach for an edit word only when you need custom literal text interspersed with digits (dates, phone numbers, currency symbols in unusual positions, or check-protection asterisks). If an edit code can express the format, use it; it is shorter and less error-prone.

Both attach to the same O-spec field line; see Chapter 10 for where the field name, end position, and edit code/word columns sit.
