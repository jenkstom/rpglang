Chapter 18 — File Access

↑ Table of contents

Summary

Beyond the automatic cycle, RPG II provides explicit file-access operations for keyed and positioned reading of DISK files. CHAIN performs random access by key (or relative record number). SETLL positions a file at a lower-bound key. READ reads the next record sequentially; READE reads the next record only if its key matches. These operations work on files declared as full-procedural (designation F) — the cycle does not read them automatically; you control access entirely. This chapter covers keyed file declaration, each operation, and the typical patterns: random lookup, sequential scan within a key range, and read-until- unequal.

Outline

18.1 The "you already know" analogy

  • CHAIN is like a dictionary lookup — dict[key] or a SQL SELECT WHERE key = value. You provide a key, you get the record (or "not found").
  • SETLL + READ is like a cursor-based scan — position at a starting point, then iterate forward. Like seek() followed by read() in file I/O.
  • READE is like a filtered cursor — read only while the key matches. Like SELECT WHERE key = value returning multiple rows.

18.2 Declaring a keyed file

  • The F-spec must declare the file as indexed (I in col 32), full-procedural (F in col 16), with key length (cols 29–30) and key start position (cols 35–38).
  • Example: 
         FSRFILE  IF  F      80  2AI     1 DISK
    This declares SRFILE as Input, Full-procedural, indexed, 80-byte records, 2-byte key starting at position 1, key type A (zoned).

18.3 CHAIN — random access by key

  • CHAIN KEY FILE searches the file for the record matching factor 1 (the key). If found, the record is read into the input buffer and fields are extracted. If not found, the no-record indicator (cols 54–55) turns on.
  • Without a key, CHAIN reads by relative record number (RRN).
  • The analogy: dict[key] or SELECT * FROM file WHERE key = value.

18.4 SETLL — set lower limit

  • SETLL KEY FILE positions the file at the first record whose key is greater than or equal to factor 1. No record is read — just positioning. Like seek() or a SQL cursor open.

18.5 READ — sequential read

  • READ FILE reads the next record from the positioned file. Sets the EOF indicator (cols 58–59) at end-of-file. Like cursor.next() or readLine().

18.6 READE — read equal key

  • READE KEY FILE reads the next record only if its key equals factor 1. If the next record's key differs (or EOF is reached), the EOF/unequal indicator (cols 58–59) turns on.
  • Combined with SETLL, this lets you scan all records matching a key — like SELECT WHERE key = value returning multiple rows.

18.8 Update files

  • An update file (F-spec type U) opens for read-write. After CHAIN or READ, the record can be rewritten in place. The O-spec drives the update: type U rewrites the last-read record; ADD in cols 16–18 appends a new record; DEL marks the record deleted.
