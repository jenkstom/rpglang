Chapter 18 — File Access

↑ Table of contents

In this chapter

Beyond the automatic cycle, RPG II provides explicit file-access operations for keyed and positioned reading of DISK files. CHAIN performs random access by key (or relative record number). SETLL positions a file at a lower-bound key. READ reads the next record sequentially; READE reads the next record only if its key matches. These operations work on files declared as full-procedural (designation F) — the cycle does not read them automatically; you control access entirely. This chapter covers keyed file declaration, each operation, and the typical patterns: random lookup, sequential scan within a key range, and read-until-unequal.

18.1 The "you already know" analogy

The cycle (see Chapter 3) reads a primary file for you, record by record. But many programs need to fetch a record on demand — look up a customer master while processing a transaction, or scan a range of records. That is what these operations are for.

This is familiar ground. CHAIN is a dictionary lookup — dict[key] or a SQL SELECT … WHERE key = value. You give a key, you get the record (or "not found"). SETLL + READ is a cursor-based scan — seek() then read(), or opening a cursor and stepping forward. READE is a filtered cursor — read only while the key matches, like SELECT WHERE key = value returning multiple rows.

18.3 CHAIN — random access by key

CHAIN KEY FILE searches the file for the record whose key matches factor 1. If found, the record is read into the input buffer and that file's fields are extracted. If not found, the no-record indicator in cols 54–55 (the HI slot) turns ON.

The CHAIN line: factor 1 TKEY at cols 18–21, operation CHAIN at cols 28–32, factor 2 CMASTER at cols 33–39, and the no-record indicator 99 at cols 54–55. The next line is conditioned on N99 (cols 9–11: N + the indicator) — it only adds MVAL when the CHAIN succeeded (indicator 99 OFF). With a master holding keys 01, 02, 03 and values 10, 20, 30, a transaction key of 02 finds the record and RPGRET becomes 20; a key of 04 misses and turns indicator 99 ON, so the add is skipped.

18.5 SETLL — set the lower limit

SETLL KEY FILE positions the file at the first record whose key is greater than or equal to factor 1. No record is read — it only sets the position, like seek() or opening a cursor. You then read with READ or READE. Factor 1 holds the key; factor 2 the file:

STKEY at cols 18–22, SETLL at cols 28–32, SRFILE at cols 33–38. After this line, the next read from SRFILE starts at or after the key in STKEY.

18.6 READ and READE — sequential reads

READ FILE reads the next record from the positioned file, regardless of key. At end-of-file, the indicator in cols 58–59 (the EQ slot) turns ON. Like cursor.next() or readLine().

READE KEY FILE reads the next record only if its key equals factor 1. If the next record's key differs, or end-of-file is reached, the indicator in cols 58–59 turns ON — the read either failed or moved past the matching group. Combined with SETLL, this scans every record sharing a key:

SKEY at cols 18–21, READE at cols 28–32, SRFILE at cols 33–38, and the end/unequal indicator 99 at cols 58–59. Each successful READE brings in the next matching record; when the key changes or EOF is hit, indicator 99 turns ON.

Walk the logic. SETLL STKEY SRFILE positions the file at key STKEY (say, 02). The label RDL marks the top of the loop. READE SKEY SRFILE reads the next record if its key equals SKEY; when it does not (or at EOF), indicator 99 turns ON. 99 GOTO DONE jumps out of the loop when the indicator is set. Otherwise ADD 1 CNT counts the matching record and GOTO RDL loops back. At end-of-file, LR fires and Z-ADD CNT RPGRET copies the count to the return value. With SRFILE holding keys 02, 02, 03, a scan for key 02 counts two records.

18.8 Resulting indicators

Each operation reports its outcome through the resulting-indicator columns on its own C-spec line. CHAIN uses the HI slot; READ and READE use the EQ slot:

OperationIndicator slotColsTurned ON when
CHAINHI54–55No record found (key/RRN miss)

READEEQ58–59End-of-file or next key unequal
SETLL——Positioning only; no indicator

Remember the C-spec resulting-indicator layout (see Chapter 9): HI at cols 54–55, LO at 56–57, EQ at 58–59. That is why the CHAIN indicator sits at 54–55 while the READ/READE indicator sits at 58–59 — they occupy different slots of the same three-group field.

18.9 Update files

An update file (F-spec type U at col 15) opens for read-write. After a CHAIN or READ fetches a record, it can be rewritten in place. The O-spec drives the update: a record line of type U rewrites the last-read record, while the operation columns can request an ADD (append a new record) or DEL (mark the record deleted). The fetch-modify-rewrite pattern — CHAIN a record, change a field, let an update O-spec write it back — mirrors the load-alter-store cycle of any mutable store.
