Chapter 13 — Subroutines and CAS

↑ Table of contents

In this chapter

RPG II subroutines — BEGSR, ENDSR, and EXSR — are reusable code blocks, the closest thing the language has to functions. They take no parameters, return no value, and have no local variables: they share the program's global fields and indicators. The CASxx operation provides a case-select mechanism — it compares two values and calls a subroutine if the comparison holds, functioning like a modern switch/case that dispatches to different subroutines. This chapter covers defining and calling subroutines, the CAS group pattern, and the constraint that shapes everything: RPG II forbids recursion.

13.1 The "you already know" analogy

A subroutine is a void function with no parameters. You define it, you call it by name, and when it finishes, control returns to the line after the call. The modern equivalent of a subroutine body is simply:

void doCalculate() {
    result = a * b;
}

You call it with EXSR DOCALC — like doCalculate().

Three things you do not get. Unlike the functions you are used to, an RPG II subroutine has no parameters, no return value, and no local variables. All communication happens through the program's global fields and indicators. A subroutine reads and writes the same fields the main code does — there is no argument list and no private scope. Think of it as a function that only uses globals.

You can call a subroutine only when a certain flag is on, or only at total time (by tagging the line with a control level in cols 7–8). The timing of the call determines when the subroutine's logic runs within the program cycle.

13.5 No recursion

RPG II forbids recursion. A subroutine cannot call itself, directly or indirectly. This is enforced at compile time: if SUBA calls SUBB and SUBB calls SUBA, the compiler rejects it. There is no call stack in the generated code — each EXSR is a simple jump-and-return, so re-entrant calls would corrupt state.

This is familiar ground — as a constraint. In Python, C#, or Java, recursion is natural: a function calls itself, and each call gets its own stack frame. RPG II subroutines are non-reentrant: they share one copy of every field, so a recursive call would overwrite the caller's data. Any problem you would solve with recursion in a modern language must be solved iteratively here — typically with a DO or DOWxx loop (see Chapter 12).

13.6 GOTO restrictions inside subroutines

Inside a subroutine, GOTO and TAG work normally — you can jump around within the subroutine body. But a GOTO may not cross the BEGSR/ENDSR boundary. You cannot jump into a subroutine from the main body, nor jump out of a subroutine to the main body. The subroutine boundary is a hard barrier for GOTO.

This restriction exists because subroutines use a return-address mechanism: ENDSR returns to the caller. A GOTO that jumped out would skip the ENDSR and leave the return address dangling. If you need to exit a subroutine early, use GOTO to a TAG just before the ENDSR — that way the subroutine still returns properly.

13.7 Subroutines and the cycle

An EXSR call inherits the timing of the line it appears on. If the EXSR is at detail time (blank control level), the subroutine runs once per record. If it is tagged LR (cols 7–8), it runs once at end-of-file. If it is tagged L1, it runs when a control break fires. See Chapter 3 for the cycle timing and Chapter 9 for the control-level column.

A common pattern: put complex per-record processing in a subroutine and call it from a detail-time line. This keeps the main calculation section clean — the detail calcs read as a sequence of high-level EXSR calls, and the messy logic is tucked away in subroutine blocks at the bottom.

Common pitfalls

  • No parameters, no locals. A subroutine cannot declare its own variables. If you need to pass information in, set a global field before the EXSR and have the subroutine read it. This is the "pseudo-parameter" pattern — verbose but the only option.
  • Subroutines go at the bottom. All BEGSR/ENDSR blocks must come after the last detail and total calculation line. Putting a subroutine definition in the middle of the main calc section will cause it to be treated as part of normal execution.
  • No recursion — direct or indirect. The compiler detects cycles in the call graph and rejects them. Use iterative loops instead.
  • GOTO stays inside. A GOTO inside a subroutine can only target a TAG within the same subroutine. To return early, jump to a TAG just before ENDSR.
