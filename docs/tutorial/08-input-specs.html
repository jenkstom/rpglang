Chapter 8 — Input Specifications (I-specs)

↑ Table of contents

In this chapter

The I-spec (column 6 = I) describes the layout of input records: which byte positions map to which named fields, what their data types and decimal positions are, and (for multi-format files) how to identify which record type was read. You declare the layout once; the program cycle extracts the fields into named variables automatically every time it reads a record. There is no parsing code to write.

8.1 The "you already know" analogy

In a modern language you might define a record layout and then write parsing code to populate it:

struct Transaction {
    int    key;       // bytes 1-2
    int    amount;    // bytes 3-5
    string name;      // bytes 6-15
};

// then, for each record:
Transaction rec;
rec.key    = atoi(raw.substr(0, 2));
rec.amount = atoi(raw.substr(2, 3));
rec.name   = raw.substr(5, 10);

In RPG II, the I-spec is the struct definition and the parser combined. You declare byte positions and field names on I-spec lines; the cycle extracts fields into named global variables automatically. No substr, no atoi, no manual parsing.

You already know this. Think of the I-spec section as a schema: "bytes 1–2 are an integer called KEY, bytes 3–5 are an integer called AMT, bytes 6–15 are a string called NAME." The runtime applies that schema to every incoming record, so by the time your C-specs run, the fields are already populated.

8.2 Two line types: record-identification and field-description

Every input file needs at least two kinds of I-spec line:

  • Record-identification line — names the file and (for multi-format files) how to recognise the record type. It carries the filename in cols 7–14 and a record-identifying indicator in cols 19–20.
  • Field-description lines — one per field. Each declares the field's byte range, name, data type, and optional control-level, matching-field, or field-indicator tags. The filename columns (7–14) are blank — these lines belong to the record-id line above them.

A file with one record type needs exactly one record-id line followed by one field line per field. Here is a complete I-spec section for a transaction file with four fields:

....+....1....+....2....+....3....+....4....+....5....+....6....+....7....+....8
     ITRANS   AA  01
     I                                        1   20KEY
     I                                        3   50AMT
     I                                        6  112SALARY
     I                                       13  20 NAME

Line by line: the first line declares that file TRANS has a record type identified by indicator 01 (cols 19–20). The next four lines map byte positions to field names. KEY occupies bytes 1–2 with 0 decimals; AMT occupies bytes 3–5 with 0 decimals; SALARY occupies bytes 6–11 with 2 decimals; NAME occupies bytes 13–20 and is alphameric (no decimal position).

8.3 Field-description line column map

Each field-description line uses the same column layout:

FieldColsDescription
Packed/Binary encoding43P (packed), B (binary), blank (zoned)
Field From44–47Starting byte position in the record (1-based)
Field To48–51Ending byte position
Decimal Positions520–9 for numeric; blank = alphameric
Field Name53–58The field's name (must start with a letter)
Control Level59–60L1–L9 (assigns this field as a control-break key)
Matching Fields61–62M1–M9 (assigns this field as a match key for multifile)
Field Indicators65–70Positive (65–66) / Negative (67–68) / Zero (69–70)

The column that beginners most often misplace is decimal positions (col 52). If you put the decimal count one column to the left or right, the field becomes the wrong type or the name shifts. Always line up against the ruler.

8.4 Numeric versus alphameric — the decimal-position column

Column 52 is the type discriminator. A digit (0–9) makes the field numeric with that many decimal positions. A blank makes it alphameric (a character string). This is the single most important column on a field-description line, because it determines which operations are valid on the field.

Compare two fields — one numeric, one alphameric — side by side:

....+....1....+....2....+....3....+....4....+....5....+....6....+....7....+....8
     I                                        6  112SALARY       ← numeric: 2 decimals
     I                                       13  20 NAME         ← alphameric: blank at col 52

For SALARY: field-from is 6 (right-justified at col 47, within the 44–47 range), field-to is 11 (right-justified at cols 50–51, within 48–51), decimal positions is 2 at col 52, and the name SALARY starts at col 53.

For NAME: field-from is 13, field-to is 20, col 52 is blank (alphameric), name starts at col 53. The blank at col 52 is what makes this a character field rather than a number.

8.5 Packed, binary, and zoned encoding (col 43)

By default, numeric fields are zoned — each byte holds one ASCII digit. Column 43 lets you override the encoding for fields stored in packed-decimal or binary format on disk:

Col 43EncodingLayout
blankZoned (default)One ASCII digit per byte
PPacked-decimalTwo BCD digits per byte, sign nibble in the last byte
BBinaryBig-endian integer (2-byte int16 or 4-byte int32)

The runtime decodes packed and binary fields into the same scaled-integer representation as zoned fields (see Chapter 5), so once a field is extracted, your C-specs treat it identically regardless of encoding. The encoding only matters for how bytes are laid out in the input file.

8.6 Control level (cols 59–60)

Assigning L1–L9 to a field marks it as a control-break key. When this field's value changes between records, the corresponding control level turns ON and total-time processing fires — letting you print subtotals whenever a group changes.

....+....1....+....2....+....3....+....4....+....5....+....6....+....7....+....8
     I                                        1   20DEPT  L1          ← L1 control break on DEPT

Here L1 sits at cols 59–60. Whenever DEPT changes value from one record to the next, indicator L1 turns on and any C-specs or O-specs conditioned on L1 execute. See Chapter 15 for the full control-level mechanism.

8.7 Matching fields (cols 61–62)

Assigning M1–M9 to a field marks it as a match key for multifile processing. When primary and secondary files both have an M1 field, the cycle merges them by key order rather than processing them independently. See Chapter 19 for the full matching-records mechanism.

8.8 Field indicators (cols 65–70)

Three indicator slots that fire at read time based on the field's value. These are resulting indicators — the cycle sets them automatically when it extracts the field:

ColumnsConditionIndicator turns ON when…
65–66Positivethe numeric field is greater than zero
67–68Negativethe numeric field is less than zero
69–70Zerothe numeric field is zero (or, for alphameric, all blanks)

You already know this. Think of field indicators as validators that fire automatically when a record is parsed. Instead of writing if (amt > 0) flag = true; after each read, you put an indicator number in cols 65–66 and the cycle does it for you. Later C-specs can then condition on that indicator.

....+....1....+....2....+....3....+....4....+....5....+....6....+....7....+....8
     I                                        3   50AMT         40  50

Here indicator 40 (cols 65–66) turns ON when AMT is positive, and indicator 50 (cols 69–70) turns ON when AMT is zero. Any C-spec conditioned on 40 or 50 then reacts to the sign of AMT without a separate COMP.

8.9 Multiple record types

A single input file can contain multiple record types — for example, header records and detail records interleaved in the same stream. You distinguish them with record-identification codes (cols 21–41 on the record-id line): three 7-column sets, each specifying a byte position, a match character, and a comparison type.

When a record is read, its identification codes are matched against the buffer. The matching type's record-identifying indicator (cols 19–20) turns ON; records matching no type are skipped. A field's field-record-relation (cols 63–64) ties it to a specific record type — it is only extracted when that type's indicator is on.

You already know this. This is like a tagged union or polymorphic deserializer: a type-discriminator byte determines which set of fields to parse. In RPG II the discriminator is a character at a known position, and the "which variant" decision happens at read time via the record-identifying indicator.

8.10 Look-ahead fields

A ** in cols 19–20 of a record-identification line marks the following field lines as look-ahead fields. These are decoded from the next (uncommitted) record rather than the current one. This lets you peek at the next record's key before the cycle processes it — useful for detecting the last record in a group before total-time processing fires. At end-of-file, look-ahead fields fill with 9s.

8.11 Putting it together

Here is a realistic I-spec section for a payroll file. The file has a 3-digit department code (control break at L1), a 5-digit employee ID, a salary with 2 decimal places, and a 15-character name:

....+....1....+....2....+....3....+....4....+....5....+....6....+....7....+....8
     IPAYROLL AA  01
     I                                        1   30DEPT  L1
     I                                        4   80EMPID
     I                                        9  152SALARY
     I                                       16  30 NAME

When a record like 03 00417 027500JOHNSON, ROBERT arrives, the cycle extracts five global fields automatically:

  • DEPT = 3 (bytes 1–3, 0 decimals, tagged L1)
  • EMPID = 417 (bytes 4–8, 0 decimals)
  • SALARY = 275.00 (bytes 9–15, 2 decimals)
  • NAME = "JOHNSON, ROBERT" (bytes 16–30, alphameric)

Every C-spec and O-spec in the program can now reference these names directly. No parsing code, no struct population, no conversion — the I-spec declaration did it all.

What's next

You now know how to declare files (F-specs) and describe their record layouts (I-specs). The remaining specification types are where your program's logic lives. The C-spec — the calculation specification — is the workhorse: every arithmetic operation, comparison, loop, and subroutine call is one C-spec line, with operands slotted into fixed columns.
