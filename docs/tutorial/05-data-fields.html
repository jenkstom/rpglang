Chapter 5 — Data Types and Fields

↑ Table of contents

In this chapter

RPG II has two fundamental data types: numeric fields (signed values with a fixed number of decimal places) and alphameric (character) fields (fixed-length byte strings). There are no floating-point types, no dynamic strings, and no user-defined types. Numeric fields are stored internally as scaled integers — the field's decimal-position count determines where the implied decimal point sits, and all arithmetic is exact. This chapter covers the numeric and alphameric data models, field length and decimal positions, the packed/binary/zoned encodings that govern how data is laid out on disk, and the sign-overpunch convention for embedding a sign in a character string.

5.1 Numeric fields — the scaled-integer model

In the languages you know, you choose among integer, floating-point, and decimal types. RPG II offers only one numeric model: fixed-point decimal. Think of it as a decimal(p, s) type that is always exact — there is no floating point anywhere in the language, so there are no floating-point rounding surprises.

This is familiar ground. A numeric field is a fixed-point decimal number: a fixed number of total digits, a fixed number of those after the decimal point, and an optional sign. The only thing to learn is how RPG II stores it.

Every numeric field has two defining attributes:

  • Length — the total number of digits (1–15), excluding the sign.
  • Decimal positions — how many of those digits fall after the implied decimal point (0–9).

Internally, values are stored as scaled integers. The stored integer equals the true value multiplied by 10 raised to the number of decimal positions. A field with length 5 and 2 decimal positions stores the value 123.45 as the integer 12345. The decimal point is not stored anywhere — it is implied by the field's decimal-position count, and the compiler inserts it whenever it formats output or aligns arithmetic.

The payoff of this model is that all arithmetic is exact. Because there is no binary fraction involved, 0.10 + 0.20 is exactly 0.30 — never 0.30000000000000004. The trade-off is a hard ceiling of 15 digits, and the programmer's responsibility to size result fields large enough to hold intermediate values (there is no automatic bignum).

Decimal positions are declared in column 52 of the I-spec (0–9, or blank for alphameric). The next chapter (Chapter 6) shows how the compiler uses these counts to align decimal points between operands automatically.

5.2 Alphameric (character) fields

The second data type is the alphameric field — a fixed-length string of bytes. This is your character data: names, addresses, codes, free text.

Why "alphameric"? IBM coined this term (a blend of alphabetic and numeric) to describe fields that can hold any character — letters, digits, symbols — without the arithmetic semantics of a true numeric field. It is the period-correct RPG II word for what modern languages call a character or string field (and what COBOL calls alphanumeric). The IBM reference manuals use "alphameric" exclusively, so we keep it here to match the source documentation you will encounter outside this tutorial.

This is familiar ground. An alphameric field is a fixed-length character array — a char[N] or a fixed-size byte buffer — not a dynamic, growable string. Its length is its width in bytes, and it never changes.

An alphameric field is defined by its length alone (the number of byte positions it spans in the record). There are no decimal positions — in fact, the way you tell RPG II a field is alphameric rather than numeric is to leave column 52 blank. A numeric field always has a digit (often 0) in column 52; an alphameric field has a blank there.

5.3 Where fields are defined

Unlike a modern language where you declare variables in one place, RPG II lets you define a field in either of two locations — and once defined, the field is globally visible everywhere. There is no local scope.

1. The I-spec (input fields). This is the primary site. An I-spec field-description line maps a range of byte positions in an input record to a field name, and gives its length and decimal positions. For example, a 3-digit numeric field AMT at record positions 1–3 with 0 decimals:

So in the line above, the 1 in cols 44–47 is the start position, the 3 in cols 48–51 is the end position, the 0 in col 52 means zero decimals (numeric), and AMT in cols 53–55 is the name. Field-description lines always follow a record-identification line, which names the file and record type — the full I-spec is covered in Chapter 8.

2. The C-spec result field (work fields). When a calculation writes to a result-field name that did not come from an I-spec, you define that field inline on the same line. 5.4 Disk encodings: zoned, packed, and binary

So far we have talked about a field's logical value (a scaled integer or a byte string). But when numeric data is stored in a file on disk, those digits have to be laid out as actual bytes. RPG II supports three encodings, and the I-spec's column 43 selects which one a field uses. The compiler handles all conversion automatically — you just declare the encoding, and the field always appears as a normal scaled integer inside your program.

EncodingCol 43LayoutSign handling
ZonedblankOne byte per digit (ASCII characters)Separate or overpunched on the last digit
PackedPTwo digits per byte (BCD), plus a sign nibbleSign nibble at the end (F = +, D = −)
BinaryBA 2- or 4-byte big-endian integerTwo's complement

Zoned (the default)

Leave column 43 blank and the field is zoned: each digit is one ASCII character. The value 12345 occupies five bytes — '1', '2', '3', '4'', '5'. This is the simplest encoding and the default; it is what you get when a file is plain text. You can even edit a zoned file by hand in a text editor.

Packed (column 43 = P)

Write P in column 43 and the field is packed decimal: two digits per byte in BCD (binary-coded decimal), with a sign nibble (half-byte) at the end. The value 12345 (five digits) packs into three bytes, because five digit-nibbles plus one sign nibble make six nibbles = three bytes:

Packed decimal is used for dense numeric storage — it roughly halves the byte cost of zoned data while staying exact (no binary-fraction conversion). The sign nibble is conventionally F for positive and D for negative.

Binary (column 43 = B)

Write B in column 43 and the field is a binary integer — a 2-byte or 4-byte big-endian value (think int16 or int32). This is the most compact for large values. A binary field is still a scaled integer logically; the encoding only changes how the raw bytes are laid out on disk.

This is familiar ground. These three encodings are just serialization formats — like choosing between plain-text, BCD, and raw binary for how you lay numbers into a byte stream. RPG II decodes them into the same scaled-integer representation internally, so your calculations never care which encoding a field uses. You declare it once on the I-spec and forget it.

5.5 Sign overpunch

There is one more encoding convention you will meet, and it comes from the punched-card era: sign overpunch. When a numeric value is carried in a character string (a zoned or alphameric field), there is no separate byte for a + or − sign. Instead, the sign is piggy-backed onto the last digit: that final character encodes both the digit value and the sign.

The mapping uses letters in place of the last digit:

DigitPositive characterNegative character
1AJ
2BK
3CL
4DM
5EN
6FO
7GP
8HQ
9IR
00 (plain)}

Plain digits 0–9 in the last position mean a positive value. The letters A–I mean positive digits 1–9, and J–R mean negative digits 1–9. So the string 000N decodes to the value −5: the three leading zeros give 000, the N supplies digit 5 with a negative sign, yielding 0005 negated. The string 123A decodes to +1231: digits 1, 2, 3, then A supplies digit 1 positively.

You rarely encode or decode overpunch by hand. The MOVE operation between a character field and a numeric field applies overpunch automatically in both directions (covered in Chapter 14). The reason to know the table is so you can read legacy data files and debug why a value that "looks" alphabetic is actually a signed number.

This is familiar ground. Sign overpunch is a compact serialization trick: stash the sign in the last digit's character code instead of spending a whole byte on it. It is the character-string analogue of packed decimal's sign nibble — both fold the sign into the data to save space.
