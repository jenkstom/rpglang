In this chapter

Before a single line of code appears, this chapter gets you oriented. You'll learn what RPG II is — a declarative, business-reporting language — why it looks so unlike anything you've used (it was shaped by 80-column punched cards and batch processing), and the three paradigm shifts you must internalise to be productive: column-based source, the implicit program cycle, and indicators. The goal is to move you past the "this looks like nothing I've ever seen" reaction and into a frame of mind where RPG II's design choices click into place.

1.1 What is RPG II?

RPG stands for Report Program Generator, and the name is the mission statement: this is a language purpose-built to make generating business reports easy. If you have ever written a program that reads a file record by record, accumulates running totals, groups results by some key, and prints a neatly formatted summary, then you have already written the kind of program RPG II was designed to make almost trivial. The difference is one of effort: in a general-purpose language you build all of that machinery by hand, whereas RPG II folds the read-loop, the grouping, the accumulation, and the report formatting into the language itself.

Where RPG II fits in the language landscape

RPG II is not a general-purpose scripting language. You would not reach for it to build a web server, a desktop GUI, a game, or a command-line tool that waits for interactive input. What it excels at is structured batch data processing and reporting: reading a large file of business records, transforming and summarising them, and producing a printed report or an output file — all unattended, from start to finish.

Think of it as a report generator with full arithmetic and control flow, not "another Python." It has the tools you would expect — variables, arithmetic, conditionals, loops, subroutines, arrays, keyed file access — but they are organised around the read-record → process → output-report pattern rather than around interactive or object-oriented programming. The vocabulary is unfamiliar, but the capabilities map onto ideas you already know.

You already know the pattern. A modern developer who writes an overnight job that pulls records from a database, computes per-region sales totals, and emails a formatted spreadsheet has written a batch report. RPG II is the language that was built so you could express that exact job declaratively instead of wiring up every stage yourself.

A short note on lineage (why the strangeness exists)

RPG II descends from the mainframe and minicomputer era of the 1960s and 70s, when source code was punched onto 80-column cards. Each card was one line of program; each column position was a physical punch slot. This single historical fact explains most of RPG II's apparent weirdness: because the format was shaped by the physical card, meaning lives in fixed column positions, not in tokens separated by whitespace.

You do not need any of this history to be productive. But understanding why columns exist makes the format feel systematic rather than arbitrary — and we return to the mechanics in Chapter 2. For now, hold one idea: position equals meaning.

1.2 The reporting mindset

Most programs you write in a modern language are interactive or event-driven: they start up, wait for input — a keystroke, an HTTP request, a timer — respond to it, and loop back to wait again. The program's lifetime is a conversation with a user or another system.

RPG II programs work differently. They are batch programs. A typical run looks like this:

  1. The program starts.
  2. It reads an entire file of records, one at a time, processing each as it goes.
  3. It produces output — a printed report, an output file, or both.
  4. When the input is exhausted, it terminates.

There is no waiting for the user, no menu, no prompt. The program runs unattended from beginning to end. This batch mindset is the lens through which every feature in this tutorial should be viewed: what part of the batch-report job does this feature automate?

This is familiar ground. Imagine a script that opens a delimited data file and reads it row by row; groups the rows by a key such as department or region; computes subtotals and a grand total per group; formats the results into a tidy summary with aligned columns, currency symbols, and zero-suppressed numbers; and writes that summary to a file or a printer.

In a modern language you would write all of that by hand: the read-loop, the grouping logic, the accumulator variables, the formatting code, and the output code. Each piece is your responsibility.

In RPG II, you declare these things and the language supplies the machinery. You say which file to read, and it supplies the read-loop. You say which fields live in a record, and it extracts them automatically. You say which field to subtotal and which key to group on, and it tracks the breaks and accumulates the totals. You describe what a report line should look like, and it formats and emits it. Your job shrinks from "build the entire pipeline" to "describe each stage, then supply only the logic unique to this report."

That shift — from building the report pipeline to describing it — is the reporting mindset in a single sentence. Keep it in mind; every chapter that follows is filling in one piece of that description.

1.3 The three paradigm shifts

These are the single most important ideas in the entire tutorial. Every chapter after this one assumes you have begun to internalise them. You do not need to master them yet — you just need to recognise them when they appear. The diagram below is the visual anchor we will keep returning to.

These three ideas are not minor syntactic quirks. They are the foundation of how RPG II thinks. Let's introduce each one in turn.

Shift 1 — Column-based source

In the languages you know, you write an assignment and the compiler splits it into tokens by whitespace. You can indent freely, add spaces for readability, and break up expressions however you like. The compiler does not care where on the line a token sits — only that the tokens appear in the right order.

In RPG II, the meaning of every character is determined by its column position on the line. A character at column 28 can mean something entirely different from the same character at column 33. There are no braces, no assignment operators, no semicolons. Instead, column 6 tells you what kind of line it is — a file declaration, an input mapping, a calculation, or an output description — and columns 7 onward hold fixed-width fields at known offsets.

This will feel alien at first, but it is completely systematic: once you learn which column ranges mean what, you can read any line the way you read a printed form, where each box on the form has a dedicated purpose. We cover the exact layout in Chapter 2.

You already know the idea of positional meaning — think of a fixed-width data record where bytes 1–6 are a customer number and bytes 7–14 are a date. RPG II applies that same principle to its own source code.

Shift 2 — The implicit program cycle

In a modern language, when you process a file you explicitly write the loop: a construct that tests whether more records remain, reads the next one, and decides when to stop. You own that loop — you write it, you control it, and you are responsible for getting the boundary conditions right.

In RPG II, you declare a file as input, and the language automatically:

  • reads the next record,
  • extracts the fields you mapped,
  • runs your calculation code on that record, and
  • loops back to read the next one — until the file is exhausted.

You never write the read-loop. You write only the body — the logic that should run for each record. The loop itself is invisible, supplied by the language. This is called the program cycle, and it is the single most important concept in all of RPG II. We devote Chapter 3 to it.

You already know the shape: it is the iterate-over-records loop you have written many times. The shock is that in RPG II this loop isn't written — it's implied. You fill in the body; the framework provides the iteration.

Shift 3 — Indicators

In a modern language, when you need to remember a yes/no fact — "was the record found?", "is this value negative?", "should this line print?" — you declare a boolean variable, set it when the fact becomes known, and test it later in a conditional.

In RPG II, there are no boolean variables to declare. Instead there is a flat, global pool of indicators: numbered boolean flags (01 through 99) plus several with special names. They exist automatically — you never declare them. You use them in two ways:

  • Conditioning. You gate an operation so it runs only when certain indicators are ON or OFF — the RPG II equivalent of an if condition.
  • Resulting. Certain operations set indicators as a side effect, recording what happened — for example, a comparison might turn on a "higher" indicator, a "lower" indicator, or an "equal" indicator.

This shared, global-flag model takes getting used to, but it is powerful once it clicks: the same flag that an operation sets can be tested by later operations and by the output section, wiring logic and output together without extra variables. We cover it in depth in Chapter 4.

You already know boolean flags and if-tests. The shock is that in RPG II the flags live in a single pre-declared global pool rather than being local variables, and many operations set them for you as a built-in result.
