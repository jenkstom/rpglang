Chapter 9 — Calculation Specifications (C-specs)

↑ Table of contents

In this chapter

The C-spec (column 6 = C) is where the program's logic lives. Every calculation — arithmetic, comparison, loop, subroutine call, field move, file access — is one C-spec line, with its components fixed by column position: control level, conditioning indicators, factor 1, operation, factor 2, result field, length/decimals, half-adjust, and resulting indicators. This chapter is a reference anatomy of the C-spec, walking every column. It also serves as the master operation index — a catalog of every operation code with its operand requirements, cross-referencing the chapter that covers each in depth.

9.1 The "you already know" analogy

A C-spec line is a single statement — like one line of a modern program. But instead of free-text syntax parsed by a tokenizer, every component lives at a fixed column offset. There are no braces, no keywords separated by whitespace, and no semicolons. Meaning is positional.

This is familiar ground. Think of each C-spec line as a single assembly instruction: opcode src1 src2 → dest, with optional condition flags and result flags. You compose multi-step calculations by chaining lines top to bottom — exactly the way you would sequence statements, just without infix operators. The order of C-spec lines is (usually) the order of execution.

9.3 Control level (cols 7–8) — when the line runs

Columns 7–8 determine when the line executes in the program cycle. This is the column that has no direct modern equivalent — it is how RPG II separates per-record logic from end-of-group and end-of-file logic without if statements:

CodeTimingWhen it fires
blankDetail timeOnce per record, during normal processing
L1–L9Total timeWhen that control level (or a higher one) is ON — i.e., when a control-break field changed
LRLast-record timeOnce, at end-of-file, after all records are processed
SRSubroutineOnly inside a BEGSR/ENDSR block — excluded from normal cycle execution
AN / ORContinuationExtends the previous line's conditioning indicators with AND/OR logic

For most lines, cols 7–8 are blank — the line runs at detail time, once per record. A line tagged L1 runs only when a control break fires; a line tagged LR runs only at end-of-file. See Chapter 3 for the cycle timing and Chapter 15 for control levels.

9.4 Conditioning indicators (cols 9–17) — the "if" guard

Three 3-column groups, ANDed together, gate whether the line executes at all. Each group can hold an indicator number (optionally prefixed with N for negation):

col:  9 10 11 | 12 13 14 | 15 16 17
      [N] II   | [N] II   | [N] II

If all three groups are blank, the line is unconditional. If any group has an indicator, the line only executes when that indicator is in the required state. 10 means "execute if indicator 10 is ON"; N20 means "execute if indicator 20 is OFF."

Here indicator 20 occupies cols 10–11 (the first group), negated by the N at col 9. The line reads: "if indicator 20 is OFF, add TAX into TOTAL." This is RPG II's equivalent of if (!ind20) total += tax; — but expressed entirely by column position. See Chapter 4 for the full indicator system.

9.5 Factor 1, Operation, Factor 2, Result (cols 18–48)

The four operand slots form the heart of every C-spec:

  • Factor 1 (cols 18–27) — the first operand. Can be a field name, a numeric literal, a character literal (quoted), or a label (for TAG/EXSR/BEGSR). Optional for many operations.
  • Operation (cols 28–32) — the opcode. Determines what happens. Always begins at col 28.
  • Factor 2 (cols 33–42) — the second operand. Required for most operations. Can be a field name, literal, array reference (ARRAY,INDEX), or file name.
  • Result field (cols 43–48) — the destination. For arithmetic, where the answer goes. For EXSR/CASxx, the subroutine name. For SETON/SETOF, blank.

A COMP operation demonstrates all four slots. Factor 1 is the left operand, Factor 2 is the right operand, and the result is recorded in the resulting indicators (cols 54–59) rather than a result field:

AMT sits at cols 18–20 (Factor 1), COMP at cols 28–31, 4 at col 33 (Factor 2), and 20 at cols 54–55 (the HI resulting indicator). The line means: "compare AMT to 4; turn indicator 20 ON if AMT > 4." See Chapter 11 for the full COMP semantics.

9.6 Result length, decimals, and half-adjust (cols 49–53)

When a result field is used for the first time — meaning it was not declared on an I-spec — you define its attributes inline here:

  • Field Length (cols 49–51) — total digits for numeric, or byte count for alphameric (1–256).
  • Decimal Positions (col 52) — 0–9 for numeric; blank for alphameric. This is the same column as on I-specs.
  • Half-Adjust (col 53) — H enables rounding. See Chapter 6.

Here is a realistic line that uses almost every column. We will decode it piece by piece:

ColsContentMeaning

9–11 10AND: indicator 10 must be ON
15–17N20AND: indicator 20 must be OFF
18–27blankFactor 1 omitted: TOTAL is both source and destination (accumulator)
28–30ADDOperation: addition

Read as a sentence: "If indicator 10 is ON and indicator 20 is OFF, add TAX into TOTAL. Turn indicator 20 ON if the result is positive." That is a lot of logic for one line — but each piece lives at a predictable column, so once you know the layout, you can read any C-spec at a glance.

For every operation, Factor 1, Factor 2, and the result field mean the same thing — they occupy the same columns. What changes is which are required, what the operation computes, and which resulting indicators it sets. The chapter listed above for each operation covers those details.

9.11 Execution order and the cycle

Within the same control level, C-specs execute top to bottom in source order. There is no operator precedence because there are no expressions — each line does one operation, and the sequence of lines is the sequence of execution.

The cycle separates C-specs by timing: detail-time specs (blank control level) run during detail processing, once per record; total-time specs (L1–L9, LR) run during total processing, when a control break fires or at end-of-file. See Chapter 3 for the full cycle description.

Subroutine specs — lines inside BEGSR/ENDSR tagged SR at cols 7–8 — are excluded from normal execution order. They run only when called via EXSR or CASxx. Subroutines always sit after all detail and total calculations in the source.

What's next

You have now seen three of the four specification types: F-specs (files), I-specs (input layout), and C-specs (calculations). The remaining spec is the O-spec — the output specification — which describes what the program produces. Where C-specs are imperative ("do this calculation"), O-specs are declarative: "when these conditions hold, print this line with these fields at these positions."

That is Chapter 10 — Output Specifications.
