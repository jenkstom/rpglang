Chapter 20 — Control and Auto Report Specifications

↑ Table of contents

In this chapter

Throughout this tutorial you have met the four core spec types — F (File), I (Input), C (Calculation), and O (Output) — plus the E-spec (Extension) that declares arrays and tables. Those five carry every line of real logic. RPG II defines two more form types that appear at the very top of a source file: the H-spec (Control) and the U-spec (Auto Report Option). They do not describe records, fields, calculations, or output. Instead they set program-wide switches (H) or hand control to a source preprocessor (U). This short chapter introduces both, explains when you will see them, and notes exactly what this compiler does with each.

20.1 The H-spec — Control Specification

"What are this program's global settings?"

You already know this as compiler flags or a configuration block — the kind of thing you pass with -D, -O2, or an environment variable in the languages you know. The H-spec is RPG II's in-source equivalent: a single line at the top of the program that sets a handful of whole-program options.

The Control Specification (form type H in column 6) is the first specification in the program when present. It is entirely optional: if you omit it, the compiler behaves as though you had written a blank H-spec, and every field keeps its documented default. At most one H-spec is meaningful; if several appear, the last one wins.

Most of its columns are switches that mattered on the original System/36 — alternative collating sequences, file translation, forms alignment for the 1P indicator, inverted printing. This compiler parses and retains every documented column but only two of them change the generated code:

Column(s)FieldEffect in this compiler
18 Currency symbol The character recognised as the floating currency symbol in edit words (default $). See the example below.
75–80 Program identification Stamps the generated module's name; blank falls back to the default RPGOBJ.

The remaining columns — date format (19), date-field separator (20), inverted print (21), alternative collating sequence (26), inquiry message (37), 1P forms position (41), file translation (43), nonprint bypass (45), and transparent literal (57) — are accepted without complaint but have no effect on the compiled program. Wiring them in is future work; none of them has an analog elsewhere in this batch compiler to plug into.

Example: changing the currency symbol

Edit words (covered in Chapter 17) can contain a floating currency symbol that travels left across the zero-suppressed digits. By default that symbol is $. Column 18 of the H-spec overrides it. Here a program declares the pound-style # symbol, then uses it in an edit word so that the value 1234 prints as #1234:

....+....1....+....2....+....3....+....4....+....5....+....6....+....7....+....8
     H           #
     FEDATA   IP  F    80              DISK
     FQOUT    O   F   132             PRINTER
     IEDATA       AA
     I                                     1   1   0DUMMY
     C                     Z-ADD1234      VAL      0
     OQOUT    D  1
     O                         VAL     20   '#0000'

The # at column 18 of the H-spec line is what makes the # inside the edit word '#0000' behave as a currency marker rather than a literal character. Without the H-spec, the same edit word would default to $ and print $1234.

Reserved currency characters. A few characters (0 * , & . - C R) already have a special meaning in edit words or edit codes and cannot serve as the currency symbol. Using one of them in column 18 produces a warning and falls back to $.

For the full column-by-column layout, see the H-spec card in the appendix.

20.2 The U-spec — Auto Report Option Specification

"Generate the boilerplate for me."

You already know this as a code generator or scaffolding tool — something that reads a short description and emits the verbose, repetitive source you would otherwise write by hand. The U-spec was RPG II's built-in version of that idea.

The Auto Report Option Specification (form type U in column 6) is the entry point of an Auto Report source program. In the original IBM toolchain a U-spec, together with the D- and O-specs it implies, was fed to a separate Auto Report preprocessor that expanded a bare field list into ordinary F/I/C/O-specs — deriving headings, spacing, and field placement automatically — before the RPG compiler proper ever ran.

This compiler does not implement Auto Report. Expanding option specs into full specifications is a large, separate preprocessing feature, and silently ignoring U lines would be worse than refusing them: an Auto Report program typically has no ordinary O-specs of its own (the option specs stand in for them), so compiling it as-is would silently produce a program with no meaningful output. Instead, every U line is a hard compile error with a clear diagnostic telling you to rewrite the program using explicit F/I/C/O-specs.

....+....1....+....2....+....3....+....4....+....5....+....6....+....7....+....8
     U ...   (an Auto Report option spec)

If you encounter a U-spec… the program was written for the IBM Auto Report preprocessor. To compile it with this tool, run the original preprocessor first (or expand the option specs by hand into explicit specifications). See the U-spec note in the appendix.

What's next

That completes the set of specification types this compiler recognises. The four core specs (F, I, C, O) carry your program's logic; the E-spec declares its arrays and tables; the H-spec sets program-wide options like the currency symbol; and the U-spec, while rejected, is documented here so you recognise it when porting older source. For a single-page visual reference of every spec type, turn to the appendix cheat sheet.
