Chapter 11 — Comparisons and Conditional Logic

↑ Table of contents

In this chapter

Every program needs to make decisions. RPG II gives you two ways to do that, and this chapter covers both. The first is the indicator-based pattern: a COMP operation compares two values and records the outcome in indicators, and then later lines are conditioned on those indicators. This is the classic RPG II style and, before structured operations were added, it was the only way to branch. The second is the structured family — IFxx, ELSE, and END — which feels much closer to the if/else you already know. We cover COMP's three-way comparison, the six IFxx operators, nested blocks, and — most importantly — when to reach for each approach.

11.1 The "you already know" analogy

You already know if/else from every language you have ever used. RPG II offers two routes to the same destination, and the diagram below shows them side by side. On the left, the structured IFxx approach wraps a block of code in a test — almost identical to what you are used to. On the right, the COMP+indicators approach splits the decision across independent lines: one line does the comparison and stores the result in a flag, and the following lines test that flag.

Why two approaches? The COMP+indicators pattern came first — it is baked into the column layout and works everywhere, including output specs where there is no IFxx. The structured IFxx operations were added later for readability. Most real programs use both: IFxx for localized if-then-else logic, and COMP when the comparison result needs to be reused or tested in an output spec.

11.2 COMP — three-way comparison

The COMP operation compares Factor 1 (cols 18–27) to Factor 2 (cols 33–42) and records the outcome in up to three resulting indicators — one for each possible result. This is the set-then-test pattern from Chapter 4, now applied to an explicit comparison.

The three resulting-indicator slots each cover one outcome:

SlotColsTurned ON when
HI54–55Factor 1 > Factor 2
LO56–57Factor 1 < Factor 2
EQ58–59Factor 1 = Factor 2

Only the matching indicator turns ON; the other two turn OFF. So a single COMP line gives you a complete three-way branch — like a compareTo() that writes its result into global flags instead of returning an integer.

11.3 IFxx — structured if-then

The IFxx operation opens a conditional block that runs only when the comparison holds. It is the closest RPG II gets to a modern if statement. The block extends until a matching ELSE or END:

Factor 1 (A, col 18) and Factor 2 (B, col 33) are the operands. The two letters appended to IF select the comparison operator — they live in the same operation columns (28–32), so IFGT is one five-character token occupying cols 28–31.

OperationMeaningModern equivalent
IFGTGreater thanif (a > b)
IFLTLess thanif (a < b)
IFEQEqualif (a == b)
IFNENot equalif (a != b)
IFGEGreater or equalif (a >= b)
IFLELess or equalif (a <= b)

This is familiar ground. IFGT A B is if (a > b). The then-branch is everything between IFGT and ELSE (or END if there is no ELSE). The whole construct is just spelled across fixed columns instead of inside parentheses and braces.

A worked trace: if A = 5 and B = 3, then A > B is true, so the then-branch runs and WIN gets 1. The else-branch is skipped. Swap the values to A = 2 and the else-branch runs instead, setting WIN to 2.

11.4 ELSE and END

ELSE marks the start of the else-branch. It is optional: if you omit it, the block is a simple if-then with no alternative. END closes the IFxx block. There is no separate ENDIF — the same END operation also closes loops and CAS groups (covered in Chapter 12 and Chapter 13).

IFxx blocks can nest. An IFxx inside an IFxx needs its own END, exactly the way nested braces work in a modern language. The compiler matches each END to the innermost open block:

Here QTY > 0 is tested first. Only if that holds does the inner QTY > 100 test run. The first END closes the inner IF; the second closes the outer one. Read it as:

if (qty > 0) {
    if (qty > 100) { tier = 1; }
    else           { tier = 2; }
}

11.5 Combining IFxx with conditioning indicators

An IFxx line is itself a C-spec, so it can carry conditioning indicators in cols 9–17 (see Chapter 4). This lets you build compound conditions: "if indicator 10 is ON and A > B, run this block." The indicator gate and the comparison are ANDed together:

If indicator 10 is OFF, the entire block is skipped — the comparison A > B is never even evaluated. If indicator 10 is ON, the comparison proceeds normally.

For OR logic between two comparisons, do not try to nest IFxx. Instead, either repeat the body on two lines (each gated by a different comparison) or use the COMP+indicator idiom: do two comparisons that SETON a shared indicator, then condition the real work on that one flag. The conditioning columns on a single line are always ANDed — there is no inline OR.

11.6 When to use which approach

  • Use IFxx when the conditional logic is localized — a clean if-then-else block that does not need to be tested anywhere else. It is more readable for modern programmers and keeps the branches visually grouped.
  • Use COMP + indicators when:
    • You need to test the same comparison result in multiple, non-contiguous places. The indicator persists, so you set it once and read it wherever you like.
    • You are conditioning output specs. O-specs use indicators (cols 23–31), not IFxx — there is no structured-operation syntax in output specifications. To make a field or line print conditionally, you must drive it from an indicator.
    • You need a three-way branch (greater / less / equal) rather than a binary true/false. COMP gives you all three outcomes in one line; IFxx would need two separate tests.

In practice, most programs use both approaches together. A typical pattern: use COMP to classify each record (setting indicators that the O-specs read), and use IFxx for any localized calculation logic that never touches output.

(In a real program, AMT would arrive from an input record, and these lines would run once per record.)

Common pitfalls

  • The three resulting slots are contiguous. Cols 54–55, 56–57, and 58–59 abut each other. Writing 30 31 32 with spaces between them puts each indicator in the wrong slot. Each indicator is exactly two columns wide with no separator — write 303132.
  • Only one COMP outcome is ON at a time. COMP turns the matching indicator ON and the other two OFF. Do not assume an old indicator value survives a COMP that did not name it — if you put an indicator in a slot, that slot is reset every time.
  • Every IFxx needs an END. A missing END swallows everything after it into the then-branch. When nesting, count your END lines — one per open IFxx.
  • No IFxx in output specs. If you need to print conditionally, set an indicator in the calculation specs and let the O-spec test it. There is no structured-operation syntax in O-specs.
