Chapter 19 — Multifile Processing

↑ Table of contents

Summary

When a program declares a primary file plus one or more secondary files, RPG II runs a multifile cycle instead of the single-file cycle. Each file's records are read into their own buffer, and each cycle iteration selects one record to process. Files tagged with a match field (M1 on the I-spec) are merged in ascending key order; the MR (matching record) indicator turns on when records from different files share the same key. Files without a match field are processed by priority (primary first). This chapter covers the multifile cycle, match-field declaration, the MR indicator, and the practical pattern of merging a primary transaction file with a secondary master file.

Outline

19.1 The "you already know" analogy

  • Multifile processing is RPG II's built-in merge join. You declare two or more files, tag a shared key field with M1 in each, and the cycle merges them by key — like SQL's SELECT ... FROM primary JOIN secondary ON primary.key = secondary.key, or like a merge-join algorithm.
  • The MR indicator tells you "this record from file A has a matching record currently held from file B" — like detecting that a join condition is satisfied for the current row.

19.2 Declaring primary and secondary files

  • One file is declared primary (designation P in F-spec col 16); additional input files are declared secondary (designation S).
  • When the cycle detects multiple input files, it switches from the single-file cycle to the multifile cycle.
  • Each file gets its own record buffer. Only the selected record's fields are extracted per iteration.

19.3 Match fields (M1)

  • To merge files by key, tag the key field in each file with M1 on the I-spec (cols 61–62). This marks it as a match field.
  • The cycle compares M1 values across files. Files with M1 are processed in ascending key order; ties (same key in multiple files) are resolved by priority — primary first, then secondaries in F-spec order.
  • Files without an M1 field are processed by priority alone (primary fully, then secondaries in order) — no key-based merging.

19.4 The MR indicator

  • When the selected record's M1 key equals another held record's M1 key, the MR indicator turns ON for that iteration.
  • This lets you write calcs that fire only when a match exists between files — e.g. "if MR is on, look up the secondary record's value and combine it with the primary's."
  • The analogy: MR is the join condition's boolean result — if (join_matched) { ... }.

19.5 Record-identifying indicators

  • Each file has a record-identifying indicator (I-spec cols 19–20). When a record from a particular file is selected, that file's indicator turns ON for the iteration. This lets you condition calcs on "which file this record came from."
  • Example: indicator 01 for primary, 02 for secondary. 01 ADD VAL RPGRET adds only when the primary record is selected; 02 ADD SVAL RPGRET adds only for secondary.

19.6 How the multifile cycle selects records

  • Each file holds its "current" record in its own buffer (with a got/valid flag).
  • Each iteration, the cycle selects one record to process:
    • Among M1 files: the record with the lowest key (ties → primary first).
    • Among non-M1 files: the primary is fully exhausted first, then secondaries in order.
  • The selected record is copied into the shared input buffer, and only that file's fields are decoded. The rest of the cycle (total/detail calc and output) is identical to the single-file path.

19.7 Look-ahead fields

  • In the multifile cycle, look-ahead fields (I-spec ** lines) peek at the next uncommitted record from a file. This is how the cycle knows whether a match will occur on the next iteration. At EOF, look-ahead fields fill with 9s.
