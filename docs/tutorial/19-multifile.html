Chapter 19 — Multifile Processing

↑ Table of contents

In this chapter

When a program declares a primary file plus one or more secondary files, RPG II runs a multifile cycle instead of the single-file cycle. Each file's records are read into their own buffer, and each cycle iteration selects one record to process. Files tagged with a match field (M1 on the I-spec) are merged in ascending key order; the MR (matching record) indicator turns on when records from different files share the same key. Files without a match field are processed by priority (primary first). This chapter covers the multifile cycle, match-field declaration, the MR indicator, and the practical pattern of merging a primary transaction file with a secondary master file.

19.1 The "you already know" analogy

The single-file cycle (see Chapter 3) reads one file for you. Multifile processing extends that to several files at once, merging them by a shared key under the cycle's control.

This is familiar ground. Multifile processing is RPG II's built-in merge join. You declare two or more files, tag a shared key field with M1 in each, and the cycle merges them by key — like SQL's SELECT … FROM primary JOIN secondary ON primary.key = secondary.key, or a merge-join algorithm. The MR indicator tells you "this record's key matches a record currently held from another file" — the join condition's boolean result.

19.3 Match fields (M1)

To merge files by key, tag the key field in each file with M1 on its I-spec line (cols 61–62). This marks it as a match field. The cycle then compares M1 values across files and processes records in ascending key order; ties (the same key in several files) are resolved by priority — primary first, then secondaries in F-spec order. The match-field code M1 sits at cols 61–62, just to the right of the field name (cols 53–58). KEY in the primary and SKEY in the secondary are both M1, so the cycle compares them to decide merge order and matching. The value fields (VAL, SVAL) at positions 3–4 carry no match code — only the join key does.

Match field vs control level. Both M1 (cols 61–62) and L1–L9 (cols 59–60) are markings on an I-spec field line, and they sit right next to each other. They are unrelated: an L-level detects a change within one file for subtotal breaks (see Chapter 15); an M-field declares the join key across files for merging. A field can carry either or both.

19.4 The MR indicator

When the selected record's M1 key equals a record currently held from another file, the MR indicator turns ON for that iteration. This lets you write calcs that fire only when a match exists between files — "if this primary transaction has a matching secondary master record, enrich it." MR is the join condition's boolean result: if (join_matched) { … }.

19.6 How the multifile cycle selects records

Each file holds its "current" record in its own buffer. Each iteration the cycle selects one record to process:

  • Among M1 files: the record with the lowest key wins; ties go to the primary first, then secondaries in F-spec order.
  • Among non-M1 files: the primary is exhausted fully first, then secondaries in order — no key-based merging.

The selected record is copied into the shared input buffer and only that file's fields are decoded. The rest of the cycle (total/detail calc and output) is identical to the single-file path.

19.7 A complete multifile merge

Here is the whole program. A primary transaction file MRPRIM (keys 1, 2, 3; values 10, 20, 30) is merged with a secondary master file MRSEC (keys 2, 4; values 5, 5). Record indicators 01 and 02 route each record's value into the running total, and MR adds 1 whenever a match occurs:

The cycle merges the two files by M1 key in this order:

  1. key=1 (P, VAL=10): indicator 01 fires, RPGRET += 10 = 10. No secondary record has key 1, so MR is off.
  2. key=2 (P, VAL=20): indicator 01 fires, RPGRET += 20 = 30. The secondary holds key 2 — a match — so MR fires, RPGRET += 1 = 31.
  3. key=2 (S, SVAL=5): indicator 02 fires, RPGRET += 5 = 36.
  4. key=3 (P, VAL=30): indicator 01 fires, RPGRET += 30 = 66. MR off (no secondary key 3).
  5. key=4 (S, SVAL=5): indicator 02 fires, RPGRET += 5 = 71. MR off.

Tally: the primary VALs (10+20+30 = 60) plus the secondary SVALs (5+5 = 10) plus one MR match (+1) gives RPGRET = 71. One program, two files, one merge — with no explicit loop, no key comparison, and no JOIN keyword in your code. The cycle and the M1 markings did all of it.

19.8 Look-ahead fields

To merge correctly, the cycle must sometimes peek at the next uncommitted record from a file before deciding which record to select this iteration. That peek uses look-ahead fields, declared under an I-spec line whose record-identification columns (19–20) hold **. Look-ahead fields read the next record's bytes without advancing the main cursor; at end-of-file they fill with 9s. You rarely write look-ahead fields yourself — the cycle uses them internally to drive the match — but they are how it knows whether a match will occur on the coming iteration.
