Chapter 4 — Indicators

↑ Table of contents

In this chapter

Indicators are RPG II's most distinctive mechanism and the one that feels most alien to a modern programmer. An indicator is a global, persistent boolean flag — you never declare it; it simply exists. Indicators serve two roles simultaneously: conditions (gate whether a calculation runs, like an if test) and results (operations set them to record outcomes, like a function return value). There are 99 general indicators (01–99) plus special-named ones (LR, L1–L9, OA–OG, and others). This chapter covers what indicators are, how to turn them on and off, how to use them to condition (gate) operations, and how operations set them as results. Mastering indicators is the key to reading and writing RPG II fluently.

4.1 What is an indicator?

Imagine a programming language where, instead of declaring bool found = false; and later writing if (found) { … }, you are handed a pre-existing pool of 99 boolean slots numbered 01 through 99. You cannot declare new ones and you cannot rename them — they exist by convention, ready to use. To turn slot 10 on you write SETON 10; to turn it off you write SETOF 10; and to run a calculation only when slot 10 is on you write the indicator's number in a fixed set of columns on that calculation line. That is an indicator.

This is familiar ground — almost. An indicator is a boolean flag, just like any bool you have used. The difference is entirely in how you spell it: there is no declaration, no assignment operator, and no if keyword. Instead the flag's number is written into fixed columns, and the column you write it in determines whether you are testing the flag (conditioning) or setting it (resulting).

Three properties make indicators behave differently from the boolean variables you are used to:

  • Global. Every calculation line can see and modify every indicator. There is no scope, no locality, and no passing of arguments — indicator 47 means the same flag everywhere.
  • Persistent. An indicator stays ON or OFF until something explicitly changes it. It does not reset at the end of a loop iteration, a subroutine call, or a cycle. Whatever value an indicator holds at the end of one record carries straight through to the start of the next.
  • Two roles. The same indicator can be tested as a condition on one line and set as a result on another. A single flag number threads through your program as both input and output.

That last property — two roles, one mechanism — is the heart of indicators, and the diagram below shows both sides of it. On the left, an indicator in the conditioning columns gates a calculation; on the right, an operation in the resulting columns records its outcome. The same numbered flag does both jobs.

4.2 The indicator pool

Although there are 99 general-purpose indicators, RPG II reserves certain indicator names for built-in purposes. You can use the general ones (01–99) for anything; the named ones are wired to specific events by the language. You will meet most of the named indicators in later chapters — the table below is a map so you know what the acronyms mean when you encounter them.

General01–99Free-form boolean flags — use these for your own logic.

For this chapter and the next several, you only need the general indicators (01–99) and LR. The rest are listed so the acronyms do not surprise you later.

4.3 Turning indicators on and off

Two operations change an indicator's value directly: SETON turns indicators ON, and SETOF turns them OFF. This is familiar ground. SETON 47 is ind[47] = true;; SETOF 47 is ind[47] = false;. That is the entire mechanism.

Because SETON and SETOF are themselves calculations, they can be conditioned by other indicators (section 4.4). That lets you write "turn indicator 47 on only if indicator 12 is on" — one flag flipping another. We will see this compound pattern after we cover conditioning.

4.4 Conditioning indicators (gating execution)

Columns 9–17 of a C-spec hold up to three conditioning indicators. These are the if tests of RPG II: a calculation line runs only when its conditioning indicators are satisfied. The nine columns are divided into three 3-column groups (9–11, 12–14, 15–17), and the three groups are ANDed together — all non-blank groups must hold for the line to execute.

Each group is written as [N]II: an optional N for negation, followed by the 2-character indicator ID. With no negation, the indicator is written in the first two columns of its group:

Count the columns carefully. On the first line, 10 occupies columns 9–10 (the first group, 9–11). On the second line, N10 occupies columns 9–11: the N in column 9, the 10 in columns 10–11. On the third line, 10 is in group 1 (9–10) and 20 is in group 2 (12–13), so the line runs only when both are on.

This is familiar ground. The three ANDed groups are a compound boolean condition — if (ind[10] && ind[20]) { … } — but expressed by position, not by syntax. There is no && token; the AND is implicit in the fixed column layout.

One important limit: there is no OR within a single line. The three groups are always ANDed. To express "indicator 10 OR indicator 20", either repeat the calculation on two lines (each conditioned on one indicator) or have both conditions SETON a shared indicator and then condition the real work on that shared flag.

4.5 Resulting indicators (recording outcomes)

The mirror image of conditioning is resulting: columns 54–59 of a C-spec hold up to three indicators that an operation sets to record what happened. They are arranged in three fixed 2-column slots: HI (54–55), LO (56–57), and EQ (58–59). Which slot means what depends on the operation:

  • COMP — compares Factor 1 to Factor 2. Sets HI if F1 > F2, LO if F1 < F2, EQ if equal.
  • ADD/SUB/MULT/DIV — set HI if the result is positive, LO if negative, EQ if zero.
  • LOKUP — sets HI/LO/EQ based on whether a higher, lower, or exact array element was found (Ch. 16).
  • CHAIN/READ — use the HI/LO slots for no-record-found / end-of-file (Ch. 18).

4.6 The set-then-test pattern

Because indicators are both conditions and results, the most common RPG II idiom is a two-step dance: one operation sets an indicator, and the next line is conditioned on that indicator. Before structured IFxx operations existed (covered in Chapter 11), this was the only way to branch. It is still everywhere in RPG II code, so you must read it fluently.

The flow is always: compare or compute → an indicator reflects the outcome → subsequent lines test that indicator. Conceptually:

  1. COMP A B with result indicator 10 in the HI slot — sets indicator 10 ON when A > B.
  2. A line conditioned on 10 runs only in the "greater" case.
  3. A line conditioned on N10 runs only in the "not greater" case — the implicit else.

Here is that pattern as real RPG II. We set VALUE to 150, compare it against 100, and route large values into BIGTOT and everything else into SMLTOT:

Line by line: Z-ADD assigns 150 to VALUE. The COMP compares VALUE to 100 and puts indicator 10 in the HI slot (cols 54–55), so it turns ON when VALUE > 100 — which, for 150, it does. The next line (condition 10) therefore runs and adds to BIGTOT. The line after it (condition N10) is skipped. Swap the value to 50 and the two conditioned lines swap roles. This is an if/else — just spread across three lines and expressed entirely in column positions.

The key mental shift. In a modern language the test and the branches live inside one if statement. In RPG II the test is a separate line whose only job is to set an indicator, and the branches are independent lines that happen to read that indicator. The "block" structure is implicit — it exists only because the same flag number appears in the resulting column of one line and the conditioning column of the next.

4.7 Indicators on other spec types

So far we have only used indicators on C-specs, but they appear on every spec type. The same numbered flag can be set on an I-spec, tested on a C-spec, and used to condition an O-spec — all in one program. That cross-spec reach is the source of both their power and their cognitive load.

I-spec field indicators (cols 65–70). When a numeric input field is read, RPG II can automatically set indicators based on the field's sign: a "plus" indicator (cols 65–66) turns on if the field is positive, a "minus" indicator (67–68) if negative, and a "zero" indicator (69–70) if zero. You then test those indicators in your calculations — no explicit comparison needed.

O-spec conditioning indicators (cols 23–31). Output lines and individual fields can be conditioned on indicators, exactly as C-specs are. This is how you suppress a field that should only print under certain conditions, or how LR triggers the grand-total line to print once at the very end (you saw this in the Chapter 2 walkthrough).

You do not need the column details of I-specs and O-specs yet — they get their own chapters (Ch. 8 and Ch. 16–17). The takeaway is that an indicator is a single global flag, however and wherever you set or test it.

4.9 Common pitfalls

  • Indicators persist — they do not auto-reset. An indicator you turn on stays on until you explicitly turn it off. A classic bug: an indicator set during one record is still on for the next record, so a conditioned line runs when you did not expect it to. When in doubt, SETOF at the start of the cycle.
  • Conditioning columns (9–17) vs. resulting columns (54–59). Writing an indicator number in the wrong block inverts its meaning entirely: in cols 9–10 it tests the flag; in cols 54–55 it sets the flag. Always check which block you are writing in.
  • Negation lives in the group's first column. N10 means "indicator 10 is OFF", not "indicator N10" (there is no indicator N10). The N is a modifier on the number that follows it.
  • No OR on one line. The three conditioning groups are ANDed. If you need OR, use two lines or a shared indicator (section 4.4).
