Programming in RPG II — A Tutorial

A hands-on guide to the RPG II programming language, written from first principles for the modern programmer.

Who this tutorial is for

This tutorial assumes you already know how to program in Python, C#, or Java. It does not teach programming from scratch. Instead it maps the concepts you already understand — variables, types, conditionals, loops, functions, arrays, file I/O, and report generation — onto RPG II's (often unfamiliar) vocabulary and mechanics.

This audience targeting should guide all future writing and editing of every chapter in this tutorial. Specifically:

  • Use language-agnostic conceptual examples. When illustrating an idea, draw on constructs common to all three languages — e.g. "an integer variable," "a while loop," "a switch/case statement," "a function that takes no arguments," "a fixed-length record." Do not show Python, C#, or Java syntax. The reader is expected to supply the modern analogy internally.
  • Name the modern concept, then introduce the RPG II equivalent. Each new topic should open with "you already know X; RPG II does this via Y."
  • Call out the three paradigm shocks early and often. RPG II differs from modern languages in three fundamental ways, and the tutorial must keep returning to them:
    1. Column-based source. Meaning lives in fixed character positions, not in tokens separated by whitespace. There are (almost) no braces, keywords as you know them, or statement terminators.
    2. The implicit program cycle. A program does not explicitly "read a record, process it, loop." The language supplies an invisible outer loop that reads every record automatically; your code is the body of that loop.
    3. Indicators. A flat set of global, persistent boolean flags — not boolean variables you declare — drive conditional execution and record the outcome of operations.
  • Favor the conceptual over the historical. Explain what a construct does and when you would use it, not the 1960s batch-mainframe lineage, except where the history genuinely aids understanding (e.g. why columns exist).
  • Keep it practical. Every concept should connect to a realistic business-programming task: reading a file of transactions, accumulating totals, producing a printed report, looking up a value in a table.
  • Cross-reference, don't duplicate. Concepts like indicators and the cycle appear throughout; each chapter should link to its foundational chapter rather than re-explaining from scratch.

How this tutorial is organized

Partially written. Chapters in Part I, Part II, Part III, and Part IV are fully written. The remaining chapters (Part V) are still scaffolds: each contains

  1. A summary — what the chapter covers and why it matters.
  2. A detailed outline — the sections, sub-topics, and example programs that should be written into it.

The summaries and outlines are the blueprint for those chapters: they define scope, ordering, and the examples to build, so that future writing can fill each one in consistently with the completed chapters.

Suggested reading order

Newcomers: read Parts I–II in order — the cycle and indicators are prerequisites for everything else. Then read the spec chapters (Part III) before the logic chapters (Part IV), because calculations reference fields you declared in I-specs and files you declared in F-specs.

Quick reference: jump straight to the spec chapter you need (F/I/C/O) or to the operation index in Chapter 9.

Conventions used in this tutorial

    ....+....1....+....2....+....3....+....4....+....5....+....6....+....7....+....8
    Column numbers are 1-based.
  • Diagrams are embedded SVG, illustrating concepts visually — column layouts, cycle flow, indicator interactions, etc.
  • Concept callouts translate an RPG II idea into modern-programmer terms.
