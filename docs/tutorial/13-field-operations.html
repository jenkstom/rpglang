Chapter 14 — Field and String Operations

↑ Table of contents

In this chapter

RPG II's field and string operations copy data between fields, compare character strings, and test individual characters or bits. The core copy operations are MOVE (right-justified copy with padding), MOVEL (left-justified copy with padding), and MOVEA (a flat byte-level copy for bulk transfers between arrays and fields). Character comparison uses the same COMP/IFxx operations you already know, but compares byte-by-byte. Two specialized operations — TESTZ (test a character's zone) and TESTB (test individual bits) — round out the toolkit. This chapter pays close attention to how MOVE and MOVEL differ in padding direction, since that is the detail that trips up most newcomers.

14.1 The "you already know" analogy

You already know how to copy data between variables — it is just assignment. RPG II splits that into three operations because fields have fixed widths and the copy must say which end to align to:

  • MOVE is a right-justified copy — like assigning a shorter value into a fixed-width field and padding on the left. The source lands at the right edge of the destination.
  • MOVEL is a left-justified copy — like a normal string copy into a buffer, padding on the right. The source lands at the left edge.
  • MOVEA is a flat memcpy between an array and a field (or two fields), treating both as raw byte sequences.

Why two copy directions? In a modern language a string variable grows and shrinks. In RPG II every field has a fixed width chosen at declaration time. When the source is shorter than the destination, something has to fill the gap. MOVE pads on the left (so the value sits at the right); MOVEL pads on the right (so the value sits at the left). For numeric fields the pad is zeros; for character fields it is blanks.

14.2 MOVE — right-justified copy

MOVE copies Factor 2 (cols 33–42) into the result field (cols 43–48), right-aligned. If the destination is wider than the source, the left side is padded — with blanks for character fields, zeros for numeric. If the source is longer than the destination, the rightmost bytes of the source are kept and the left overflow is dropped.

....+....1....+....2....+....3....+....4....+....5....+....6....+....7....+....8
     C                     Z-ADD73        SRC
     C                     MOVE SRC       DST      ← right-justified copy
     C                     ADD  DST       RPGRET

Here MOVE is at cols 28–31, SRC at col 33 (Factor 2), and DST at col 43 (result). Both are numeric, so the copy is right-aligned and zero-padded. With SRC = 73, DST receives 73, and RPGRET ends up 73.

Type conversion. When MOVE crosses type boundaries — numeric source to character destination, or vice versa — it applies sign-overpunch encoding on the rightmost digit. This is how RPG II converts between string and numeric representations. See Chapter 5 for the overpunch details. For same-type moves (numeric to numeric, character to character), no conversion happens — just the alignment and padding.

14.3 MOVEL — left-justified copy

MOVEL is the mirror of MOVE: it copies Factor 2 into the result field left-aligned, with padding on the right. If the source is longer than the destination, the leftmost bytes are kept and the right side is truncated. Use MOVEL when you want text at the start of the field — natural reading order:

....+....1....+....2....+....3....+....4....+....5....+....6....+....7....+....8
     C                     MOVEL'WORLD'   NAME

MOVEL fills cols 28–32 completely — it is a five-character token — so Factor 2 begins immediately at col 33 with no gap. The literal 'WORLD' occupies cols 33–39, and NAME starts at col 43. After this line, NAME holds "WORLD     " — the text left-aligned, padded with blanks on the right.

Five-character opcodes pack tightly. MOVEL, MOVEA, BEGSR, ENDSR, and the xx-suffixed comparison ops (DOWLT, CASGE, etc.) all fill cols 28–32 exactly. Factor 2 then starts at col 33 with zero spaces between the opcode and the operand — e.g. MOVEL'WORLD'. This is correct column-based behavior, not a formatting error.

14.4 MOVEA — array ↔ field byte move

MOVEA does a left-justified, byte-level copy between Factor 2 and the result field, treating both as flat byte sequences. It copies as many bytes as fit in the smaller of the two. The classic use is bulk-transferring data between a character array and a single large field, or repacking fields:

....+....1....+....2....+....3....+....4....+....5....+....6....+....7....+....8
     C                     MOVEACSRC      CDST     ← byte copy CSRC → CDST
     C           CSRC      COMP CDST                     20

MOVEA fills cols 28–32; CSRC starts at col 33 (Factor 2) and CDST at col 43 (result). Both must be character (alphameric) fields. After the copy, the COMP line verifies that the two fields now hold identical bytes — if they match, indicator 20 turns ON in the EQ slot. (In a real program, CSRC and CDST would be declared on I-specs or E-specs with their character widths.)

14.5 Character comparison

COMP, IFxx, DOWxx, and the other comparison operations all work on alphameric fields as well as numeric. For character operands, the comparison is byte-by-byte, left-aligned, with the shorter operand blank-padded on the right — just like String.compareTo() in a modern language:

....+....1....+....2....+....3....+....4....+....5....+....6....+....7....+....8
     C           CNAME     COMP 'SMITH'              20

CNAME is at col 18 (Factor 1), COMP at cols 28–31, the literal 'SMITH' at cols 33–39 (Factor 2), and indicator 20 at cols 54–55 (the HI slot). After the comparison, indicator 20 is ON when CNAME > 'SMITH'. Comparing "ABC" to "ABD": they differ at the third byte, C < D, so the LO outcome holds. This is plain lexicographic ordering on the raw byte values.

14.6 TESTZ — test character zone

TESTZ examines the leftmost byte of the result field (which must be alphameric) and sets resulting indicators based on its "zone" — a leftover from the EBCDIC punched-card era, where the high nibble of the first character carried sign information. Factor 1 and Factor 2 are not used; the field to test goes in the result field (cols 43–48):

Result indicator slotColsTurned ON when the character is
HI54–55In the "plus" zone: &, A–I, a–i
LO56–57In the "minus" zone: -, J–R, j–r
EQ58–59Anything else (the "zero" zone)
....+....1....+....2....+....3....+....4....+....5....+....6....+....7....+....8
     C                     MOVE 'J'       CF    1   ← CF = 'J' (1-byte char field)
     C                     TESTZ          CF         010203

TESTZ fills cols 28–32; the field CF sits at col 43 (result), and the three indicators 01, 02, 03 occupy cols 54–59 contiguously (HI, LO, EQ). Because 'J' is in the minus zone, indicator 02 (LO) turns ON. A later line conditioned on 02 can react to that:

....+....1....+....2....+....3....+....4....+....5....+....6....+....7....+....8
     C  02                 Z-ADD2         RPGRET    ← runs because 'J' is minus zone

In practice TESTZ is rarely used in new code, but it appears in legacy programs that relied on overpunch sign conventions.

14.7 TESTB — test bits

TESTB tests individual bits within a one-byte character field. Factor 2 specifies which bits to test — either as a bit-number literal like '025' (each digit names a bit, 0 = leftmost / highest) or as a one-position character field whose ON bits form the mask. The field being tested goes in the result field (cols 43–48):

Result indicator slotColsTurned ON when
HI54–55Every tested bit is OFF
LO56–57The tested bits are mixed (some on, some off)
EQ58–59Every tested bit is ON
....+....1....+....2....+....3....+....4....+....5....+....6....+....7....+....8
     C                     MOVE 'C'       CF    1   ← CF = 'C' (ASCII 0x43)
     C                     TESTB'0'       CF         010203

TESTB fills cols 28–32; the mask literal '0' is at col 33 (Factor 2), naming bit 0 (the leftmost / highest-order bit). CF is at col 43 (result), and indicators 01/02/03 fill cols 54–59. The ASCII code for 'C' is 0x43 = 01000011 in binary — bit 0 (the 0x80 position) is OFF. Since the one tested bit is OFF, indicator 01 (HI) turns ON. A line conditioned on 01 then runs:

....+....1....+....2....+....3....+....4....+....5....+....6....+....7....+....8
     C  01                 Z-ADD1         RPGRET    ← runs because bit 0 of 'C' is off

You already know this. TESTB is a bitwise AND test. "HI if all tested bits OFF" is (val & mask) == 0; "EQ if all tested bits ON" is (val & mask) == mask; "LO" is the mixed case in between. RPG II just writes the outcome into indicators instead of a boolean expression.

Common pitfalls

  • MOVE vs. MOVEL padding direction. This is the #1 source of confusion. MOVE right-justifies (pads left); MOVEL left-justifies (pads right). If your text ends up with unexpected leading blanks, you probably used MOVE where you wanted MOVEL.
  • The field to test is in the result column. For TESTZ and TESTB, the field under examination goes in cols 43–48 (the result field) — not in Factor 2. Factor 2 holds the bit mask for TESTB and is blank for TESTZ.
  • Resulting indicators for TESTZ/TESTB are contiguous. Like COMP, the three slots at cols 54–59 abut each other. Write 010203, not 01 02 03.
  • Character comparison is byte-order, not locale-aware. COMP on alphameric fields compares raw byte values. Uppercase and lowercase sort separately because their ASCII codes differ — there is no case-insensitive comparison built in.
