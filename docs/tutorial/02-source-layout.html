Chapter 2 — The Source Layout

↑ Table of contents

In this chapter

This chapter teaches you how to look at RPG II source before teaching any semantics. RPG II source is a sequence of lines divided into fixed columns, and column 6 of each line declares its spec type: F (File), I (Input), C (Calculation), or O (Output). We'll introduce the column ruler, the four spec types at a high level, the ordering convention (F → I → C → O), and walk through a complete example program. By the end you should be able to visually parse any RPG II listing and identify what each section does — even without knowing the details yet.

2.1 The 80-column legacy

As we noted in Chapter 1, RPG II source was originally punched onto 80-column cards. Each card was one line of program; each column position was a physical punch slot. The format survived into digital source files because it was deeply embedded in the language design — the compiler was built to read fixed positions, so changing the format would have meant changing the language.

The practical consequence is the single most important thing to internalise about reading RPG II: whitespace is not a separator. You cannot freely indent, add spaces, or align things the way you can in the languages you know. A character at column 28 has a completely different meaning from the same character at column 33. The compiler does not scan for tokens delimited by spaces; it reads fixed character positions.

You already know positional meaning — think of a fixed-width data record where bytes 1–6 are a customer number and bytes 7–14 are a date. RPG II applies that same principle to its own source code. Reading an RPG II program is like reading a printed form: each box on the form has a dedicated purpose, and the value you write in that box is interpreted according to which box it is, not according to how it is spaced.

This does not mean RPG II source is chaotic. Far from it — it is extremely regular. Once you learn which column ranges mean what, every line follows the same anatomy. The rest of this chapter teaches you that anatomy.

2.2 The column ruler

Because column position is everything, every code example in this tutorial is preceded by a ruler line that helps you count positions:

....+....1....+....2....+....3....+....4....+....5....+....6....+....7....+....8

Each group is ten columns wide. The + marks every fifth column; the digit marks every tenth. Columns are 1-based — column 1 is the leftmost position. So column 15 sits directly under the 1 in ....1, and column 28 sits under the 8 in ...2.

Global column conventions

Regardless of spec type, every line in an RPG II program shares the same high-level column structure:

ColumnsPurposeRequired?
1–5 Sequence number. A purely cosmetic numbering that helped keep card decks in order. Modern source files usually leave it blank. Optional
6 Form type — the single letter that declares what kind of line this is: F, I, C, or O. This is the most important column in the entire language. Required
7–59 Spec-specific fields. The meaning of these columns depends on the form type in column 6. This is where the actual content lives, and each spec type divides this range differently. (See sections 2.3 and 2.5 below.) Varies
60–74 Comments. Free-form text appended to any line — an inline note for the human reader. Optional
75–80 Program ID. Another card-deck relic: a short identifier stamped on every card so a dropped deck could be sorted back together. Modern source files usually leave it blank. Optional

Comment lines

In addition to the inline comments in columns 60–74, you can make an entire line a comment by placing an asterisk (*) in column 7. The form-type letter in column 6 is still present (so the compiler knows which section the comment belongs to), but the asterisk tells it to ignore the rest of the line entirely:

....+....1....+....2....+....3....+....4....+....5....+....6....+....7....+....8
     C*  This entire line is a comment — the * in column 7 says "ignore me"

You will see comment lines like this in every example throughout the tutorial. They are how RPG II programmers annotate their work.

The diagram above shows a single calculation line. Column 6 holds C, declaring it a calculation. Columns 7–59 are divided into Factor 1, Operation, Factor 2, Result field, and resulting indicators — but that internal division is specific to C-specs. Each spec type slices columns 7–59 its own way, which is exactly what the rest of this chapter introduces and what Chapters 7–10 cover column by column.

2.3 The four spec types

Think of a program as answering four questions in sequence: What files am I using? What do the records look like? What do I compute? What do I output? Each spec type answers one of those questions. This maps onto things you already do in every program you write:

You already know this pattern. When you write a program that processes a file, you: (1) open the file, (2) define a structure to hold each record, (3) write the logic that transforms the data, and (4) format and emit the output. RPG II gives each of those four phases its own dedicated spec type, and each one is a different kind of declaration rather than executable code.

F-spec — File Description (col 6 = F)

"What files does this program use?"

You already know this as opening a file and specifying its mode — read, write, or update. An F-spec declares each file's name, whether it is input or output, its record format (fixed or variable), its record length, and the device it lives on (disk, printer, workstation). One F-spec line declares one file.

Here is a typical pair of F-spec lines — one declaring an input file on disk, one declaring an output file to a printer:

....+....1....+....2....+....3....+....4....+....5....+....6....+....7....+....8
     FSALES   IP  F    80              DISK
     FREPORT  O   F   132             PRINTER

Don't worry about every column yet — we cover F-specs in detail in Chapter 7. For now, just see the shape: column 6 is F, the filename sits in columns 7–14, and the device sits further right. The IP after the filename means Input/Primary (this file feeds the program cycle); O means Output.

I-spec — Input (col 6 = I)

"What does each record contain?"

You already know this as defining a struct or a class to hold a parsed record. An I-spec maps byte positions within an input record to named fields. Instead of writing code to extract substrings at offsets, you declare which columns of the record correspond to which field name, and the language extracts them for you automatically on every cycle.

Here is an I-spec that identifies records from the SALES file and maps a 3-byte numeric field at record positions 1–3 to the name AMT:

....+....1....+....2....+....3....+....4....+....5....+....6....+....7....+....8
     ISALES   AA  01
     I                                        1    3AMT

The first line identifies the record; the second line maps field positions to names. We cover I-specs in Chapter 8.

C-spec — Calculation (col 6 = C)

"What should the program do?"

You already know this as the body of your program — your assignment statements, arithmetic, if-blocks, loops, and function calls. C-specs contain the executable logic. Each line is one operation: it names a Factor 1, an operation code, a Factor 2, and a result field — arranged in fixed columns rather than as a tokenised expression.

Here are two C-spec lines — one that adds AMT into TOTAL, and one that sets the LR (last-record) indicator to signal the program cycle that it is time to finish:

....+....1....+....2....+....3....+....4....+....5....+....6....+....7....+....8
     C                     ADD  AMT       TOTAL
     C                     SETON                     LR

Notice there is no equals sign and no semicolon. The operation code ADD sits in its own fixed column range; AMT (Factor 2) and TOTAL (result) each sit in theirs. We cover C-specs in Chapter 9.

O-spec — Output (col 6 = O)

"What should the output look like?"

You already know this as printf or string-format templates, but baked into the language as structured declarations. An O-spec declaratively describes report lines: which fields print at which positions, what constants (like column headings) accompany them, and under what conditions each line should emit. You are describing the shape of the output, not writing code to produce it.

Here are O-spec lines that print a detail line showing the value of AMT, followed by a total line showing TOTAL:

....+....1....+....2....+....3....+....4....+....5....+....6....+....7....+....8
     OREPORT  D  1
     O                                   20 'Amount ='
     O                         AMT       30
     OREPORT  T  2    LR
     O                                   20 'Grand total ='
     O                         TOTAL     30

The D means a detail line (printed once per record); the T means a total line (printed once, at the end). We cover O-specs in Chapter 16.

The four spec types flow naturally from top to bottom: declare your files, describe their records, write the logic, describe the output. That is the shape of every RPG II program.

2.4 Spec ordering

Spec types appear in a fixed order: F, then I, then C, then O. This is not a style suggestion — the compiler expects the sections in this sequence, because each section builds on the one before it:

  • F-specs come first because every other section references the files declared here.
  • I-specs come next because they describe the records of the files declared in the F-specs, and the C-specs will reference the field names defined here.
  • C-specs come after, because the calculation logic operates on the fields declared in the I-specs and writes results that the O-specs will reference.
  • O-specs come last, because the output descriptions reference both the fields from I-specs and the results computed in C-specs.

Within each spec type, lines appear in the order the program needs them. Each F-spec declares one file. I-spec field-description lines follow their record-identification line. C-specs execute top to bottom. O-spec field lines follow their record line.

A program does not need all four spec types. A pure calculation program with no file I/O might have only C-specs. A program that reads a file and does no calculation might skip C-specs entirely and just map fields in I-specs directly to output in O-specs. But most real-world business programs use all four.

2.5 Your first program: a visual walkthrough

Now let's put the pieces together. Below is a complete, working RPG II program. It reads a file of sales records, adds each record's AMT field to a running TOTAL, prints each amount on a detail line, and prints the grand total at the end.

This is a preview: we are not explaining the semantics of every operation yet — that is the job of the chapters ahead. The goal right now is simply to see the shape. Try to identify each spec section by its column-6 letter before reading the annotations.

....+....1....+....2....+....3....+....4....+....5....+....6....+....7....+....8
     FSALES   IP  F    80              DISK             ← F-spec: input file
     FREPORT  O   F   132             PRINTER            ← F-spec: output file

     ISALES   AA  01                                    ← I-spec: record id
     I                                        1    3AMT                  ← I-spec: field map

     C                     ADD  AMT       TOTAL         ← C-spec: accumulate
     C                     SETON                     LR  ← C-spec: signal end

     OREPORT  D  1                                       ← O-spec: detail line
     O                                   20 'Amount ='   ← O-spec: constant
     O                         AMT       30              ← O-spec: field
     OREPORT  T  2    LR                                  ← O-spec: total line
     O                                   20 'Grand total ='  ← O-spec: constant
     O                         TOTAL     30              ← O-spec: field

Walking through each section

The F-spec section (lines 1–2). Two files are declared. SALES is an input file on disk — the IP means "input, primary," telling the language this is the main file that feeds the automatic read loop. REPORT is an output file to a printer — the O means output. Both are fixed-format (F) with the record lengths shown (80 for input, 132 for printed output).

The I-spec section (lines 4–5). The first line identifies the record: it names the file (SALES), declares the record type (AA), and assigns a record-identifying indicator (01). The second line maps record positions 1 through 3 to the field name AMT. From now on, whenever the cycle reads a record, the first three characters are automatically extracted and available as AMT.

The C-spec section (lines 7–8). Two operations. The first adds AMT into TOTAL — this runs once per record, automatically, because of the program cycle. The second uses SETON to turn on the LR indicator, which tells the cycle "after this record, stop reading and run your final total output." (LR stands for "last record"; we cover it in Chapter 3.)

The O-spec section (lines 10–15). Two output record descriptions. The first (type D = detail) prints once per record: a literal 'Amount =' ending at column 20, followed by the value of AMT ending at column 30. The second (type T = total, conditioned on LR) prints once at the very end: the literal 'Grand total =' followed by TOTAL.

What to notice

  • There is no explicit read-loop anywhere. The program cycle reads each record silently; the C-specs and O-specs are the body of that invisible loop. This is paradigm shift #2 from Chapter 1 in action.
  • There are no braces, no assignment operators, no semicolons. Every line is a fixed-column declaration; the operation code (ADD, SETON) sits in its designated column range, and the operands sit in theirs. This is paradigm shift #1.
  • The output is described, not coded. You do not write print statements — you declare what each report line looks like and when it should appear, and the language emits it for you.
  • The LR indicator weaves through all three sections: it is set in a C-spec and tested in an O-spec, without any explicit wiring. This is a first taste of paradigm shift #3 — indicators — which Chapter 4 covers in full.

You do not need to understand every detail yet. The point of this walkthrough is visual literacy — recognising the four sections and seeing how they fit together into a complete program. The chapters that follow will revisit each spec type column by column, starting with the program cycle in Chapter 3.

What's next

You can now visually parse an RPG II program: you know the column ruler, the global column conventions, the four spec types, and the F→I→C→O ordering. The biggest question remaining is the one we have been hinting at since Chapter 1: how does the read-loop actually work? If there is no explicit loop in the source, what exactly happens between records?

That is the program cycle, and it is the subject of Chapter 3. Understanding it is the single most important step in becoming productive with RPG II — everything else builds on it.
