Chapter 15 — Control Levels and Report Breaks

↑ Table of contents

Summary

Control levels (L1–L9) are RPG II's mechanism for automatic grouping and subtotalling. You tag a field on the I-spec as a control field (e.g. L1 for department code), and whenever that field's value changes between records, the cycle fires a control break: the corresponding level indicator turns ON, total-time calculations run (subtotals, accumulator resets), and total-time output prints. At LR (end- of-file), all levels turn ON so the final group's subtotals and the grand total both print. This chapter covers assigning control levels, the cascade rule (a break at L3 also fires L1 and L2), total-time execution order, and the practical pattern of accumulating in detail calcs and printing/resetting at total time.

Outline

15.1 The "you already know" analogy

  • In a modern language, producing a subtotal report means: 
    records = sorted(records, key=lambda r: r.dept)
for group, items in groupby(records, key=lambda r: r.dept):
    subtotal = sum(i.amount for i in items)
    print(f"{group}: {subtotal}")
print(f"Grand total: {grand}")
    You manually sort, group, accumulate, and print subtotals.
  • In RPG II, if the input file is already sorted by department, you tag the department field as L1 on the I-spec, accumulate in detail calcs, and write total-time calcs/output conditioned on L1 and LR. The cycle handles break detection, cascading, and timing automatically. It's built-in GROUP BY with automatic subtotal output.

15.2 Assigning control levels on I-specs

  • I-spec columns 59–60 assign a control level (L1–L9) to a field. When that field's value changes between records, the level turns ON.
  • Example: KEY L1 in cols 53–60 means "this field is a level-1 control field."
  • The input file must be sorted by the control field — RPG II detects changes, it doesn't sort.

15.3 The cascade rule

  • When a break occurs at level Ln, all lower levels also turn ON. A break at L3 fires L1, L2, and L3.
  • This means C-specs conditioned on L1 run for any break (L1 through L9), L2 runs for breaks at L2 or higher, etc.
  • Total-time calcs and output run in ascending level order: L0 (always), then L1, L2, …, L9, then LR. This ensures subtotals accumulate correctly within nested groups.
  • L0 is a special level that is always ON at total time — use it for calcs that should run at every total-time pass.

15.4 The accumulator-and-reset pattern

  • The standard subtotal pattern uses two fields:
    • SUB — the running subtotal, accumulated in detail calcs (ADD AMT SUB every record).
    • PSUB — a "print subtotal" that SUB is moved into at total time, so the printed value isn't affected by the reset.
  • Total-time calcs (conditioned on L1): MOVE SUB PSUB (save for printing), Z-ADD 0 SUB (reset for next group).
  • Total-time output (type T, conditioned on L1): prints PSUB.
  • GRAND total accumulates across all records (never reset until LR): ADD AMT GRAND in detail calcs, printed at LR total time.

15.5 LR and final subtotals

  • At EOF, LR turns ON and all L1–L9 also turn ON. This ensures the last group's subtotal prints (the last break never happens naturally because there's no "next" record with a different key).
  • Execution order at LR: total calcs/output run L0→L1→…→L9 (printing the last group's subtotals), then LR-specific calcs/output (grand totals).

15.6 First-record bypass

  • On the very first record, total time is skipped — the control-field baseline is being established, and there's no "previous value" to compare against. No spurious break fires on record 1.

15.7 Nested control levels

  • Multiple control fields create nested groups. Example: region = L1, department = L2. A change in region fires both L1 and L2; a change in department (within the same region) fires only L2.
  • The cascade rule and ascending execution order handle this correctly — inner-group (higher-level) subtotals print before outer-group subtotals.
