Chapter 15 — Control Levels and Report Breaks

↑ Table of contents

In this chapter

Control levels (L1–L9) are RPG II's mechanism for automatic grouping and subtotalling. You tag a field on the I-spec as a control field (e.g. L1 for a department code), and whenever that field's value changes between records, the cycle fires a control break: the corresponding level indicator turns ON, total-time calculations run (subtotals, accumulator resets), and total-time output prints. At LR (end-of-file), all levels turn ON so the final group's subtotals and the grand total both print. This chapter covers assigning control levels, the cascade rule, total-time execution order, and the practical pattern of accumulating in detail calcs and printing/resetting at total time.

15.1 The "you already know" analogy

In a modern language, producing a subtotal report means sorting, grouping, accumulating, and printing yourself:

records = sorted(records, key=lambda r: r.dept)
grand = 0
for group, items in groupby(records, key=lambda r: r.dept):
    subtotal = 0
    for i in items:
        subtotal += i.amount
        grand    += i.amount
        print(f"{group}  {i.amount}")
    print(f"Subtotal: {subtotal}")
print(f"Grand total: {grand}")

You write the sort, the grouping detection, the running sums, and the print calls. In RPG II, if the input file is already sorted by department, you tag the department field as L1 on the I-spec, accumulate in detail calcs, and write total-time calcs/output conditioned on L1 and LR. The cycle handles break detection, cascading, and timing automatically. It is built-in GROUP BY with automatic subtotal output.

This is familiar ground. A control break is a GROUP BY whose key-change detection and timing the language owns, not you. You only write three things: accumulate (per record), print and reset (per group), and print the grand total (at the end). The cycle decides when each of those runs.

The chapter on the program cycle (Chapter 3) introduced the total-time phase. Control levels are what make total time do useful work, so this chapter assumes you have read it.

15.2 Assigning a control level on an I-spec

A control field is just an ordinary input field with one extra marking: its control level, written in cols 59–60 of the field's I-spec line. When that field's value changes from one record to the next, the named level indicator turns ON. Recall the I-spec field columns (see Chapter 8):

Watch the columns. The to-position (col 51) and the decimal-position count (col 52) sit right next to each other, so the fragment   1  10KEY reads visually like "from 1, to 10" — but it is from 1, to 1, decimals 0. The "10" is the to-digit and the decimals-digit run together. Always anchor yourself to the ruler; meaning is positional, not token-shaped.

For the AMT field, 2 at col 47 is the from position and 3 at col 51 is the to position, so AMT occupies positions 2–3. It has no control level (cols 59–60 blank) — only the group-key field gets one.

The input file must already be sorted by the control field. RPG II detects changes between consecutive records; it does not sort for you. A file that is not sorted by KEY will produce a break on every record where KEY differs from the previous one, giving nonsense subtotals.

15.3 Detail-time: accumulate every record

Accumulation happens at detail time — once per record, in C-specs whose control-level columns (7–8) are blank. Two running sums are kept: SUB, the current group's subtotal, and GRAND, the grand total across all groups:

Both lines have blank control-level columns (7–8), so they run at detail time for every record. The operation ADD sits at cols 28–30; factor 2 AMT at cols 33–35; the result fields SUB / GRAND start at col 43. (See Chapter 9 for the full C-spec anatomy, and Chapter 6 for ADD.)

Because detail calcs run before total time, every record's AMT has already been added into SUB by the time a break fires. That is the whole trick: accumulate first, report later.

15.4 The cascade rule

When a break occurs at level Ln, all lower-numbered levels also turn ON. A break at L3 fires L1, L2, and L3. This cascade is what makes nested grouping work: a C-spec or O-spec conditioned on L1 runs for any break (L1 through L9); one conditioned on L2 runs only for breaks at L2 or higher; and so on.

Total-time calcs and output run in ascending level order: first any L0 lines (a special always-on-at-total-time level), then L1, L2, …, L9, then LR. This ordering ensures inner-group (higher-level) subtotals accumulate correctly within their outer group before the outer group itself reports.

15.5 Total-time: save, print, reset

When an L1 break fires, the total-time C-specs run. They do three things: copy the accumulated subtotal into a "print" field, then zero the accumulator so the next group starts fresh. The two-step save-and-reset is the standard idiom:

The L1 at cols 7–8 is what makes these total-time lines: they run only when an L1 break (or a higher break, or LR) fires. Why a separate PSUB? Because we print SUB in the O-spec after these calcs run — if we printed SUB directly, the reset to zero would have already clobbered it. MOVE copies SUB into PSUB; then Z-ADD 0 zeros SUB. The O-spec prints PSUB, which still holds the group total.

The accumulator-and-reset pattern. Use two fields: SUB (accumulate in detail calcs, never printed directly) and PSUB (the saved copy, printed at total time). At the break: MOVE SUB PSUB then Z-ADD 0 SUB. GRAND follows the same accumulate rule but is never reset until LR.

15.8 LR fires every level

At end-of-file, LR turns ON and all of L1–L9 also turn ON. This is not arbitrary politeness — it is essential. The last group's subtotal can never print any other way: there is no "next" record with a different key to trigger a natural break. By forcing every level ON at LR, the cycle guarantees the final group's subtotals and the grand total both print. Execution order at LR is total calcs/output L0→L1→…→L9 (printing the last group's subtotals), then LR-specific calcs/output (grand totals).

15.9 First-record bypass

On the very first record, total time is skipped. The control-field baseline is being established — there is no "previous value" to compare against — so no spurious break fires on record 1. Total-time calcs and output first become eligible from the second record onward.

15.10 Nested control levels

Multiple control fields create nested groups. For example, tag region as L1 and department as L2 on their respective I-spec field lines. A change in region fires both L1 and L2 (cascade); a change in department within the same region fires only L2. The cascade rule and ascending execution order handle the bookkeeping — inner-group (higher-level) subtotals print before outer-group subtotals. When region changes, both fire, and L2's total runs before L1's (ascending order), so the last department's subtotal is reported before the region subtotal that contains it.
